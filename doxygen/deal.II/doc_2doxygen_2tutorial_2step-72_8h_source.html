549  * <a name="Manualassembly"></a>
550  * <h5>Manual assembly</h5>
551  *
552 
553  *
554  * The assembly functions are the interesting contributions to this tutorial.
555  * The assemble_system_unassisted() method implements exactly the same
556  * assembly function as is detailed in @ref step_15 "step-15", but in this instance we
557  * use the MeshWorker::mesh_loop() function to multithread the assembly
558  * process. The reason for doing this is quite simple: When using
559  * automatic differentiation, we know that there is to be some additional
560  * computational overhead incurred. In order to mitigate this performance
561  * loss, we'd like to take advantage of as many (easily available)
562  * computational resources as possible. The MeshWorker::mesh_loop() concept
563  * makes this a relatively straightforward task. At the same time, for the
564  * purposes of fair comparison, we need to do the same to the implementation
565  * that uses no assistance when computing the residual or its linearization.
566  * (The MeshWorker::mesh_loop() function is first discussed in @ref step_12 "step-12" and
567  * @ref step_16 "step-16", if you'd like to read up on it.)
568  *
569 
570  *
571  * The steps required to implement the multithreading are the same between the
572  * three functions, so we'll use the assemble_system_unassisted() function
573  * as an opportunity to focus on the multithreading itself.
574  *
575  * @code
576  * template <int dim>
577  * void MinimalSurfaceProblem<dim>::assemble_system_unassisted()
578  * {
579  * system_matrix = 0;
580  * system_rhs = 0;
581  *
582  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
583  *
584  * @endcode
585  *
586  * The MeshWorker::mesh_loop() expects that we provide two exemplar data
587  * structures. The first, `ScratchData`, is to store all large data that
588  * is to be reused between threads. The `CopyData` will hold the
589  * contributions to the linear system that come from each cell. These
590  * independent matrix-vector pairs must be accumulated into the
591  * global linear system sequentially. Since we don't need anything
592  * on top of what the MeshWorker::ScratchData and MeshWorker::CopyData
593  * classes already provide, we use these exact class definitions for
594  * our problem. Note that we only require a single instance of a local
595  * matrix, local right-hand side vector, and cell degree of freedom index
596  * vector -- the MeshWorker::CopyData therefore has `1` for all three
597  * of its template arguments.
598  *
599  * @code
600  * using ScratchData = MeshWorker::ScratchData<dim>;
601  * using CopyData = MeshWorker::CopyData<1, 1, 1>;
602  *
603  * @endcode
604  *
605  * We also need to know what type of iterator we'll be working with
606  * during assembly. For simplicity, we just ask the compiler to work
607  * this out for us using the decltype() specifier, knowing that we'll
608  * be iterating over active cells owned by the @p dof_handler.
609  *
610  * @code
611  * using CellIteratorType = decltype(dof_handler.begin_active());
612  *
613  * @endcode
614  *
615  * Here we initialize the exemplar data structures. Since we know that
616  * we need to compute the shape function gradients, weighted Jacobian,
617  * and the position of the quadrate points in real space, we pass these
618  * flags into the class constructor.
619  *
620  * @code
621  * const ScratchData sample_scratch_data(fe,
622  * quadrature_formula,
623  * update_gradients |
624  * update_quadrature_points |
625  * update_JxW_values);
626  * const CopyData sample_copy_data(dofs_per_cell);
627  *
628  * @endcode
629  *
630  * Now we define a lambda function that will perform the assembly on
631  * a single cell. The three arguments are those that will be expected by
632  * MeshWorker::mesh_loop(), due to the arguments that we'll pass to that
633  * final call. We also capture the @p this pointer, which means that we'll
634  * have access to "this" (i.e., the current `MinimalSurfaceProblem<dim>`)
635  * class instance, and its private member data (since the lambda function is
636  * defined within a MinimalSurfaceProblem<dim> method).
637  *
638 
639  *
640  * At the top of the function, we initialize the data structures
641  * that are dependent on the cell for which the work is being
642  * performed. Observe that the reinitialization call actually
643  * returns an instance to an FEValues object that is initialized
644  * and stored within (and, therefore, reused by) the
645  * `scratch_data` object.
646  *
647 
648  *
649  * Similarly, we get aliases to the local matrix, local RHS
650  * vector, and local cell DoF indices from the `copy_data`
651  * instance that MeshWorker::mesh_loop() provides. We then
652  * initialize the cell DoF indices, knowing that the local matrix
653  * and vector are already correctly sized.
654  *
655  * @code
656  * const auto cell_worker = [this](const CellIteratorType &cell,
657  * ScratchData & scratch_data,
658  * CopyData & copy_data) {
659  * const auto &fe_values = scratch_data.reinit(cell);
660  *
661  * FullMatrix<double> & cell_matrix = copy_data.matrices[0];
662  * Vector<double> & cell_rhs = copy_data.vectors[0];
663  * std::vector<types::global_dof_index> &local_dof_indices =
664  * copy_data.local_dof_indices[0];
665  * cell->get_dof_indices(local_dof_indices);
666  *
667  * @endcode
668  *
669  * For Newton's method, we require the gradient of the solution at the
670  * point about which the problem is being linearized.
671  *
672 
673  *
674  * Once we have that, we can perform assembly for this cell in
675  * the usual way. One minor difference to @ref step_15 "step-15" is that we've
676  * used the (rather convenient) range-based loops to iterate
677  * over all quadrature points and degrees-of-freedom.
678  *
679  * @code
680  * std::vector<Tensor<1, dim>> old_solution_gradients(
681  * fe_values.n_quadrature_points);
682  * fe_values.get_function_gradients(current_solution,
683  * old_solution_gradients);
684  *
685  * for (const unsigned int q : fe_values.quadrature_point_indices())
686  * {
687  * const double coeff =
688  * 1.0 / std::sqrt(1.0 + old_solution_gradients[q] *
689  * old_solution_gradients[q]);
690  *
691  * for (const unsigned int i : fe_values.dof_indices())
692  * {
693  * for (const unsigned int j : fe_values.dof_indices())
694  * cell_matrix(i, j) +=
695  * (((fe_values.shape_grad(i, q) // ((\nabla \phi_i
696  * * coeff // * a_n
697  * * fe_values.shape_grad(j, q)) // * \nabla \phi_j)
698  * - // -
699  * (fe_values.shape_grad(i, q) // (\nabla \phi_i
700  * * coeff * coeff * coeff // * a_n^3
701  * * (fe_values.shape_grad(j, q) // * (\nabla \phi_j
702  * * old_solution_gradients[q]) // * \nabla u_n)
703  * * old_solution_gradients[q])) // * \nabla u_n)))
704  * * fe_values.JxW(q)); // * dx
705  *
706  * cell_rhs(i) -= (fe_values.shape_grad(i, q) // \nabla \phi_i
707  * * coeff // * a_n
708  * * old_solution_gradients[q] // * u_n
709  * * fe_values.JxW(q)); // * dx
710  * }
711  * }
712  * };

714  * @endcode

716  * The second lambda function that MeshWorker::mesh_loop() requires is
717  * one that performs the task of accumulating the local contributions
718  * in the global linear system. That is precisely what this one does,
719  * and the details of the implementation have been seen before. The
720  * primary point to recognize is that the local contributions are stored
721  * in the `copy_data` instance that is passed into this function. This
722  * `copy_data` has been filled with data during @a some call to the
723  * `cell_worker`.
724  *
725  * @code
726  * const auto copier = [dofs_per_cell, this](const CopyData &copy_data) {
727  * const FullMatrix<double> &cell_matrix = copy_data.matrices[0];
728  * const Vector<double> & cell_rhs = copy_data.vectors[0];
729  * const std::vector<types::global_dof_index> &local_dof_indices =
730  * copy_data.local_dof_indices[0];
731  *
732  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
733  * {
734  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
735  * system_matrix.add(local_dof_indices[i],
736  * local_dof_indices[j],
737  * cell_matrix(i, j));
738  *
739  * system_rhs(local_dof_indices[i]) += cell_rhs(i);
740  * }
741  * };
742  *
743  * @endcode
744  *
745  * We have all of the required functions definitions in place, so
746  * now we call the MeshWorker::mesh_loop() to perform the actual
747  * assembly. We pass a flag as the last parameter which states
748  * that we only want to perform the assembly on the
749  * cells. Internally, MeshWorker::mesh_loop() then distributes the
750  * available work to different threads, making efficient use of
751  * the multiple cores almost all of today's processors have to
752  * offer.
753  *
754  * @code
755  * MeshWorker::mesh_loop(dof_handler.active_cell_iterators(),
756  * cell_worker,
757  * copier,
758  * sample_scratch_data,

760  * MeshWorker::assemble_own_cells);

762  * @endcode

766  * the Newton updates @f$\delta u^n@f$.

768  * @code
769  * hanging_node_constraints.condense(system_matrix);
770  * hanging_node_constraints.condense(system_rhs);

786  * <a name="Assemblyviadifferentiationoftheresidualvector"></a>
787  * <h5>Assembly via differentiation of the residual vector</h5>
788  *
789 
790  *
791  * As outlined in the introduction, what we need to do for this
792  * second approach is implement the local contributions @f$F(U)^K@f$
793  * from cell @f$K@f$ to the residual vector, and then let the
794  * AD machinery deal with how to compute the
795  * derivatives @f$J(U)_{ij}^K=\frac{\partial F(U)^K_i}{\partial U_j}@f$
796  * from it.
797  *
798 
799  *
800  * For the following, recall that
801  * @f[
802  * F(U)_i^K \dealcoloneq
803  * \int\limits_K\nabla \varphi_i \cdot \left[ \frac{1}{\sqrt{1+|\nabla
804  * u|^{2}}} \nabla u \right] \, dV ,
805  * @f]
806  * where @f$u(\mathbf x)=\sum_j U_j \varphi_j(\mathbf x)@f$.
807  *
808 
809  *
810  * Let us see how this is implemented in practice:
811  *
812  * @code
813  * template <int dim>
814  * void MinimalSurfaceProblem<dim>::assemble_system_with_residual_linearization()
815  * {
816  * system_matrix = 0;
817  * system_rhs = 0;
818  *
819  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
820  *
821  * using ScratchData = MeshWorker::ScratchData<dim>;
822  * using CopyData = MeshWorker::CopyData<1, 1, 1>;
823  * using CellIteratorType = decltype(dof_handler.begin_active());
824  *
825  * const ScratchData sample_scratch_data(fe,
826  * quadrature_formula,
827  * update_gradients |
828  * update_quadrature_points |
829  * update_JxW_values);
830  * const CopyData sample_copy_data(dofs_per_cell);
831  *
832  * @endcode
833  *
834  * We'll define up front the AD data structures that we'll be using,
835  * utilizing the techniques shown in @ref step_71 "step-71".
836  * In this case, we choose the helper class that will automatically compute
837  * the linearization of the finite element residual using Sacado forward
838  * automatic differentiation types. These number types can be used to
839  * compute first derivatives only. This is exactly what we want, because we
840  * know that we'll only be linearizing the residual, which means that we
841  * only need to compute first-order derivatives. The return values from the
842  * calculations are to be of type `double`.
843  *
844 
845  *
846  * We also need an extractor to retrieve some data related to the field
847  * solution to the problem.
848  *
849  * @code
850  * using ADHelper = Differentiation::AD::ResidualLinearization<
851  * Differentiation::AD::NumberTypes::sacado_dfad,
852  * double>;
853  * using ADNumberType = typename ADHelper::ad_type;
854  *
855  * const FEValuesExtractors::Scalar u_fe(0);
856  *
857  * @endcode
858  *
859  * With this, let us define the lambda function that will be used
860  * to compute the cell contributions to the Jacobian matrix and
861  * the right hand side:
862  *
863  * @code
864  * const auto cell_worker = [&u_fe, this](const CellIteratorType &cell,
865  * ScratchData & scratch_data,
866  * CopyData & copy_data) {
867  * const auto & fe_values = scratch_data.reinit(cell);
868  * const unsigned int dofs_per_cell = fe_values.get_fe().n_dofs_per_cell();
869  *
870  * FullMatrix<double> & cell_matrix = copy_data.matrices[0];
871  * Vector<double> & cell_rhs = copy_data.vectors[0];
872  * std::vector<types::global_dof_index> &local_dof_indices =
873  * copy_data.local_dof_indices[0];
874  * cell->get_dof_indices(local_dof_indices);
875  *
876  * @endcode
877  *
878  * We'll now create and initialize an instance of the AD helper class.
879  * To do this, we need to specify how many independent variables and
880  * dependent variables there are. The independent variables will be the
881  * number of local degrees of freedom that our solution vector has,
882  * i.e., the number @f$j@f$ in the per-element representation of the
883  * discretized solution vector
884  * @f$u (\mathbf{x})|_K = \sum\limits_{j} U^K_i \varphi_j(\mathbf{x})@f$
885  * that indicates how many solution coefficients are associated with
886  * each finite element. In deal.II, this equals
887  * FiniteElement::dofs_per_cell. The number of dependent variables will be
888  * the number of entries in the local residual vector that we will be
889  * forming. In this particular problem (like many others that employ the
890  * [standard Galerkin
891  * method](https://en.wikipedia.org/wiki/Galerkin_method)) the number of
892  * local solution coefficients matches the number of local residual
893  * equations.
894  *
895  * @code
896  * const unsigned int n_independent_variables = local_dof_indices.size();
897  * const unsigned int n_dependent_variables = dofs_per_cell;
898  * ADHelper ad_helper(n_independent_variables, n_dependent_variables);
899  *
900  * @endcode
901  *
902  * Next we inform the helper of the values of the solution, i.e., the
903  * actual values for @f$U_j@f$ about which we
904  * wish to linearize. As this is done on each element individually, we
905  * have to extract the solution coefficients from the global solution
906  * vector. In other words, we define all of those coefficients @f$U_j@f$
907  * where @f$j@f$ is a local degree of freedom as the independent variables
908  * that enter the computation of the vector @f$F(U)^{K}@f$ (the dependent
909  * function).
910  *
911 
912  *
913  * Then we get the complete set of degree of freedom values as
914  * represented by auto-differentiable numbers. The operations
915  * performed with these variables are tracked by the AD library
916  * from this point until the object goes out of scope. So it is
917  * <em>precisely these variables</em> with respect to which we will
918  * compute derivatives of the residual entries.
919  *
920  * @code
921  * ad_helper.register_dof_values(current_solution, local_dof_indices);
922  *
923  * const std::vector<ADNumberType> &dof_values_ad =
924  * ad_helper.get_sensitive_dof_values();
925  *
926  * @endcode
927  *
928  * Then we do some problem specific tasks, the first being to
929  * compute all values, (spatial) gradients, and the like based on
930  * "sensitive" AD degree of freedom values. In this instance we are
931  * retrieving the solution gradients at each quadrature point. Observe
932  * that the solution gradients are now sensitive
933  * to the values of the degrees of freedom as they use the @p ADNumberType
934  * as the scalar type and the @p dof_values_ad vector provides the local
935  * DoF values.
936  *
937  * @code
938  * std::vector<Tensor<1, dim, ADNumberType>> old_solution_gradients(
939  * fe_values.n_quadrature_points);
940  * fe_values[u_fe].get_function_gradients_from_local_dof_values(
941  * dof_values_ad, old_solution_gradients);
942  *
943  * @endcode
944  *
945  * The next variable that we declare will store the cell residual vector
946  * contributions. This is rather self-explanatory, save for one
947  * <b>very important</b> detail:
948  * Note that each entry in the vector is hand-initialized with a value
949  * of zero. This is a <em>highly recommended</em> practice, as some AD
950  * libraries appear not to safely initialize the internal data
951  * structures of these number types. Not doing so could lead to some
952  * very hard to understand or detect bugs (appreciate that the author
953  * of this program mentions this out of, generally bad, experience). So
954  * out of an abundance of caution it's worthwhile zeroing the initial
955  * value explicitly. After that, apart from a sign change the residual
956  * assembly looks much the same as we saw for the cell RHS vector before:
957  * We loop over all quadrature points, ensure that the coefficient now
958  * encodes its dependence on the (sensitive) finite element DoF values by
959  * using the correct `ADNumberType`, and finally we assemble the
960  * components of the residual vector. For complete clarity, the finite
961  * element shape functions (and their gradients, etc.) as well as the
962  * "JxW" values remain scalar
963  * valued, but the @p coeff and the @p old_solution_gradients at each
964  * quadrature point are computed in terms of the independent
965  * variables.
966  *
967  * @code
968  * std::vector<ADNumberType> residual_ad(n_dependent_variables,
969  * ADNumberType(0.0));
970  * for (const unsigned int q : fe_values.quadrature_point_indices())
971  * {
972  * const ADNumberType coeff =
973  * 1.0 / std::sqrt(1.0 + old_solution_gradients[q] *
974  * old_solution_gradients[q]);
975  *
976  * for (const unsigned int i : fe_values.dof_indices())
977  * {
978  * residual_ad[i] += (fe_values.shape_grad(i, q) // \nabla \phi_i
979  * * coeff // * a_n
980  * * old_solution_gradients[q]) // * u_n
981  * * fe_values.JxW(q); // * dx
982  * }
983  * }
984  *
985  * @endcode
986  *
987  * Once we have the full cell residual vector computed, we can register
988  * it with the helper class.
989  *
990 
991  *
992  * Thereafter, we compute the residual values (basically,
993  * extracting the real values from what we already computed) and
994  * their Jacobian (the linearization of each residual component
995  * with respect to all cell DoFs) at the evaluation point. For
996  * the purposes of assembly into the global linear system, we
997  * have to respect the sign difference between the residual and
998  * the RHS contribution: For Newton's method, the right hand
999  * side vector needs to be equal to the *negative* residual
1000  * vector.
1001  *
1002  * @code
1003  * ad_helper.register_residual_vector(residual_ad);
1004  *
1005  * ad_helper.compute_residual(cell_rhs);
1006  * cell_rhs *= -1.0;
1007  *
1008  * ad_helper.compute_linearization(cell_matrix);
1009  * };
1010  *
1011  * @endcode
1012  *
1013  * The remainder of the function equals what we had previously:
1014  *
1015  * @code
1016  * const auto copier = [dofs_per_cell, this](const CopyData &copy_data) {
1017  * const FullMatrix<double> &cell_matrix = copy_data.matrices[0];
1018  * const Vector<double> & cell_rhs = copy_data.vectors[0];
1019  * const std::vector<types::global_dof_index> &local_dof_indices =
1020  * copy_data.local_dof_indices[0];
1021  *
1022  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1023  * {
1024  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
1025  * system_matrix.add(local_dof_indices[i],
1026  * local_dof_indices[j],
1027  * cell_matrix(i, j));
1028  *
1029  * system_rhs(local_dof_indices[i]) += cell_rhs(i);
1030  * }
1031  * };

1033  * MeshWorker::mesh_loop(dof_handler.active_cell_iterators(),
1034  * cell_worker,
1035  * copier,

1056  *
1057  * <a name="Assemblyviadifferentiationoftheenergyfunctional"></a>
1058  * <h5>Assembly via differentiation of the energy functional</h5>
1059  *
1060 
1061  *
1062  * In this third approach, we compute residual and Jacobian as first
1063  * and second derivatives of the local energy functional
1064  * @f[
1065  * E\left( U \right)^K
1066  * \dealcoloneq \int\limits_{K} \Psi \left( u \right) \, dV
1067  * \approx \sum\limits_{q}^{n_{\textrm{q-points}}} \Psi \left( u \left(
1068  * \mathbf{X}_{q} \right) \right) \underbrace{\vert J_{q} \vert \times
1069  * W_{q}}_{\text{JxW(q)}}
1070  * @f]
1071  * with the energy density given by
1072  * @f[
1073  * \Psi \left( u \right) = \sqrt{1+|\nabla u|^{2}} .
1074  * @f]
1075  *
1076 
1077  *
1078  * Let us again see how this is done:
1079  *
1080  * @code
1081  * template <int dim>
1082  * void MinimalSurfaceProblem<dim>::assemble_system_using_energy_functional()
1083  * {
1084  * system_matrix = 0;
1085  * system_rhs = 0;
1086  *
1087  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
1088  *
1089  * using ScratchData = MeshWorker::ScratchData<dim>;
1090  * using CopyData = MeshWorker::CopyData<1, 1, 1>;
1091  * using CellIteratorType = decltype(dof_handler.begin_active());
1092  *
1093  * const ScratchData sample_scratch_data(fe,
1094  * quadrature_formula,
1095  * update_gradients |
1096  * update_quadrature_points |
1097  * update_JxW_values);
1098  * const CopyData sample_copy_data(dofs_per_cell);
1099  *
1100  * @endcode
1101  *
1102  * In this implementation of the assembly process, we choose the helper
1103  * class that will automatically compute both the residual and its
1104  * linearization from the cell contribution to an energy functional using
1105  * nested Sacado forward automatic differentiation types.
1106  * The selected number types can be used to compute both first and
1107  * second derivatives. We require this, as the residual defined as the
1108  * sensitivity of the potential energy with respect to the DoF values (i.e.
1109  * its gradient). We'll then need to linearize the residual, implying that
1110  * second derivatives of the potential energy must be computed. You might
1111  * want to compare this with the definition of `ADHelper` used int
1112  * previous function, where we used
1113  * `Differentiation::AD::ResidualLinearization<Differentiation::AD::NumberTypes::sacado_dfad,double>`.
1114  *
1115  * @code
1116  * using ADHelper = Differentiation::AD::EnergyFunctional<
1117  * Differentiation::AD::NumberTypes::sacado_dfad_dfad,
1118  * double>;
1119  * using ADNumberType = typename ADHelper::ad_type;
1120  *
1121  * const FEValuesExtractors::Scalar u_fe(0);
1122  *
1123  * @endcode
1124  *
1125  * Let us then again define the lambda function that does the integration on
1126  * a cell.
1127  *
1128 
1129  *
1130  * To initialize an instance of the helper class, we now only require
1131  * that the number of independent variables (that is, the number
1132  * of degrees of freedom associated with the element solution vector)
1133  * are known up front. This is because the second-derivative matrix that
1134  * results from an energy functional is necessarily square (and also,
1135  * incidentally, symmetric).
1136  *
1137  * @code
1138  * const auto cell_worker = [&u_fe, this](const CellIteratorType &cell,
1139  * ScratchData & scratch_data,
1140  * CopyData & copy_data) {
1141  * const auto &fe_values = scratch_data.reinit(cell);
1142  *
1143  * FullMatrix<double> & cell_matrix = copy_data.matrices[0];
1144  * Vector<double> & cell_rhs = copy_data.vectors[0];
1145  * std::vector<types::global_dof_index> &local_dof_indices =
1146  * copy_data.local_dof_indices[0];
1147  * cell->get_dof_indices(local_dof_indices);
1148  *
1149  * const unsigned int n_independent_variables = local_dof_indices.size();
1150  * ADHelper ad_helper(n_independent_variables);
1151  *
1152  * @endcode
1153  *
1154  * Once more, we register all cell DoFs values with the helper, followed
1155  * by extracting the "sensitive" variant of these values that are to be
1156  * used in subsequent operations that must be differentiated -- one of
1157  * those being the calculation of the solution gradients.
1158  *
1159  * @code
1160  * ad_helper.register_dof_values(current_solution, local_dof_indices);
1161  *
1162  * const std::vector<ADNumberType> &dof_values_ad =
1163  * ad_helper.get_sensitive_dof_values();
1164  *
1165  * std::vector<Tensor<1, dim, ADNumberType>> old_solution_gradients(
1166  * fe_values.n_quadrature_points);
1167  * fe_values[u_fe].get_function_gradients_from_local_dof_values(
1168  * dof_values_ad, old_solution_gradients);
1169  *
1170  * @endcode
1171  *
1172  * We next create a variable that stores the cell total energy.
1173  * Once more we emphasize that we explicitly zero-initialize this value,
1174  * thereby ensuring the integrity of the data for this starting value.
1175  *
1176 
1177  *
1178  * The aim for our approach is then to compute the cell total
1179  * energy, which is the sum of the internal (due to right hand
1180  * side functions, typically linear in @f$U@f$) and external
1181  * energies. In this particular case, we have no external
1182  * energies (e.g., from source terms or Neumann boundary
1183  * conditions), so we'll focus on the internal energy part.
1184  *
1185 
1186  *
1187  * In fact, computing @f$E(U)^K@f$ is almost trivial, requiring only
1188  * the following lines:
1189  *
1190  * @code
1191  * ADNumberType energy_ad = ADNumberType(0.0);
1192  * for (const unsigned int q : fe_values.quadrature_point_indices())
1193  * {
1194  * const ADNumberType psi = std::sqrt(1.0 + old_solution_gradients[q] *
1195  * old_solution_gradients[q]);
1196  *
1197  * energy_ad += psi * fe_values.JxW(q);
1198  * }
1199  *
1200  * @endcode
1201  *
1202  * After we've computed the total energy on this cell, we'll
1203  * register it with the helper. Based on that, we may now
1204  * compute the desired quantities, namely the residual values
1205  * and their Jacobian at the evaluation point. As before, the
1206  * Newton right hand side needs to be the negative of the
1207  * residual:
1208  *
1209  * @code
1210  * ad_helper.register_energy_functional(energy_ad);
1211  *
1212  * ad_helper.compute_residual(cell_rhs);
1213  * cell_rhs *= -1.0;
1214  *
1215  * ad_helper.compute_linearization(cell_matrix);
1216  * };
1217  *
1218  * @endcode
1219  *
1220  * As in the previous two functions, the remainder of the function is as
1221  * before:
1222  *
1223  * @code
1224  * const auto copier = [dofs_per_cell, this](const CopyData &copy_data) {
1225  * const FullMatrix<double> &cell_matrix = copy_data.matrices[0];
1226  * const Vector<double> & cell_rhs = copy_data.vectors[0];
1227  * const std::vector<types::global_dof_index> &local_dof_indices =
1228  * copy_data.local_dof_indices[0];
1229  *
1230  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1231  * {
1232  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
1233  * system_matrix.add(local_dof_indices[i],
1234  * local_dof_indices[j],
1235  * cell_matrix(i, j));
1236  *
1237  * system_rhs(local_dof_indices[i]) += cell_rhs(i);
1238  * }
1239  * };
1240  *
1241  * MeshWorker::mesh_loop(dof_handler.active_cell_iterators(),
1242  * cell_worker,
1243  * copier,
1244  * sample_scratch_data,
1245  * sample_copy_data,
1246  * MeshWorker::assemble_own_cells);
1247  *
1248  * hanging_node_constraints.condense(system_matrix);
1249  * hanging_node_constraints.condense(system_rhs);
1250  *
1251  * std::map<types::global_dof_index, double> boundary_values;
1252  * VectorTools::interpolate_boundary_values(dof_handler,
1253  * 0,
1254  * Functions::ZeroFunction<dim>(),
1255  * boundary_values);
1256  * MatrixTools::apply_boundary_values(boundary_values,
1257  * system_matrix,
1258  * newton_update,
1259  * system_rhs);
1260  * }
1261  *
1262  *
1263  * @endcode

1405  * std::vector<Tensor<1, dim>> gradients(n_q_points);

1426  * }

1434  *

1755 <a name="Possibilitiesforextensions"></a><h3> Possibilities for extensions </h3>
1756 
1757 
1758 与步骤-71一样，有几个与自动区分有关的项目可以进一步评估。
1759 
1760 - 应调查其他AD框架的使用情况，并展望其他实施方式可能提供性能优势。
1761 
1762 - 除了本教程中硬编码的数字类型外，还值得对AD数字类型进行评估。关于在有限元水平上采用的两次微分类型，混合微分模式（"RAD-FAD"）原则上应该比这里采用的单一模式（"FAD-FAD"）类型的计算效率更高。RAD-FAD类型没有被默认选择的原因是，在撰写本文时，在Sacado库中，它的实现仍然存在一些错误，导致内存泄漏。 这在 @ref auto_symb_diff 模块中有所记载。
1763 
1764 - 也许使用低精度类型（即 "浮动"）作为AD数字的标量类型可以减少装配时的计算费用。使用 "float "作为矩阵和残差的数据类型并不是不合理的，因为牛顿更新只是为了让我们更接近解决方案，而不是实际*到解决方案；因此，考虑使用降低精度的数据类型来计算这些更新，然后将这些更新累积到使用全 "双 "精度的解决方案向量中，是有意义的。
1765 
1766 - 在装配过程中可能减少资源的另一个方法是将AD的实现作为一个构成模型。这类似于步骤71中采用的方法，并将自动微分的起点推到了计算链的上一级。这反过来意味着AD库跟踪的操作更少，从而降低了微分的成本（尽管我们会在每个单元的正交点进行微分）。

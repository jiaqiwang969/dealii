895  * <a name="Theinversepermeabilitytensor"></a>
896  * <h3>The inverse permeability tensor</h3>
897  *
898 
899  *
900  * In addition to the other equation data, we also want to use a
901  * permeability tensor, or better -- because this is all that appears in the
902  * weak form -- the inverse of the permeability tensor,
903  * <code>KInverse</code>. For the purpose of verifying the exactness of the
904  * solution and determining convergence orders, this tensor is more in the
905  * way than helpful. We will therefore simply set it to the identity matrix.
906  *
907 
908  *
909  * However, a spatially varying permeability tensor is indispensable in
910  * real-life porous media flow simulations, and we would like to use the
911  * opportunity to demonstrate the technique to use tensor valued functions.
912  *
913 
914  *
915  * Possibly unsurprisingly, deal.II also has a base class not only for
916  * scalar and generally vector-valued functions (the <code>Function</code>
917  * base class) but also for functions that return tensors of fixed dimension
918  * and rank, the <code>TensorFunction</code> template. Here, the function
919  * under consideration returns a dim-by-dim matrix, i.e. a tensor of rank 2
920  * and dimension <code>dim</code>. We then choose the template arguments of
921  * the base class appropriately.
922  *
923 
924  *
925  * The interface that the <code>TensorFunction</code> class provides is
926  * essentially equivalent to the <code>Function</code> class. In particular,
927  * there exists a <code>value_list</code> function that takes a list of
928  * points at which to evaluate the function, and returns the values of the
929  * function in the second argument, a list of tensors:
930  *
931  * @code
932  * template <int dim>
933  * class KInverse : public TensorFunction<2, dim>
934  * {
935  * public:
936  * KInverse()
937  * : TensorFunction<2, dim>()
938  * {}
939  *
940  * virtual void
941  * value_list(const std::vector<Point<dim>> &points,
942  * std::vector<Tensor<2, dim>> & values) const override;
943  * };
944  *
945  *
946  * @endcode
947  *
948  * The implementation is less interesting. As in previous examples, we add a
949  * check to the beginning of the class to make sure that the sizes of input
950  * and output parameters are the same (see @ref step_5 "step-5" for a discussion of this
951  * technique). Then we loop over all evaluation points, and for each one
952  * set the output tensor to the identity matrix.
953  *
954 
955  *
956  * There is an oddity at the top of the function (the
957  * `(void)points;` statement) that is worth discussing. The values
958  * we put into the output `values` array does not actually depend
959  * on the `points` arrays of coordinates at which the function is
960  * evaluated. In other words, the `points` argument is in fact
961  * unused, and we could have just not given it a name if we had
962  * wanted. But we want to use the `points` object for checking
963  * that the `values` object has the correct size. The problem is
964  * that in release mode, `AssertDimension` is defined as a macro
965  * that expands to nothing; the compiler will then complain that
966  * the `points` object is unused. The idiomatic approach to
967  * silencing this warning is to have a statement that evaluates
968  * (reads) variable but doesn't actually do anything: That's what
969  * `(void)points;` does: It reads from `points`, and then casts
970  * the result of the read to `void`, i.e., nothing. This statement
971  * is, in other words, completely pointless and implies no actual
972  * action except to explain to the compiler that yes, this
973  * variable is in fact used even in release mode. (In debug mode,
974  * the `AssertDimension` macro expands to something that reads
975  * from the variable, and so the funny statement would not be
976  * necessary in debug mode.)
977  *
978  * @code
979  * template <int dim>
980  * void KInverse<dim>::value_list(const std::vector<Point<dim>> &points,
981  * std::vector<Tensor<2, dim>> & values) const
982  * {
983  * (void)points;
984  * AssertDimension(points.size(), values.size());
985  *
986  * for (auto &value : values)
987  * value = unit_symmetric_tensor<dim>();
988  * }
989  * } // namespace PrescribedSolution
990  *
991  *
992  *
993  * @endcode

1002  * <a name="MixedLaplaceProblemMixedLaplaceProblem"></a>
1003  * <h4>MixedLaplaceProblem::MixedLaplaceProblem</h4>
1004  *
1005 
1006  *
1007  * In the constructor of this class, we first store the value that was
1008  * passed in concerning the degree of the finite elements we shall use (a
1009  * degree of zero, for example, means to use RT(0) and DG(0)), and then
1010  * construct the vector valued element belonging to the space @f$X_h@f$ described
1011  * in the introduction. The rest of the constructor is as in the early
1012  * tutorial programs.
1013  *
1014 
1015  *
1016  * The only thing worth describing here is the constructor call of the
1017  * <code>fe</code> variable. The <code>FESystem</code> class to which this
1018  * variable belongs has a number of different constructors that all refer to
1019  * binding simpler elements together into one larger element. In the present
1020  * case, we want to couple a single RT(degree) element with a single
1021  * DQ(degree) element. The constructor to <code>FESystem</code> that does
1022  * this requires us to specify first the first base element (the
1023  * <code>FE_RaviartThomas</code> object of given degree) and then the number
1024  * of copies for this base element, and then similarly the kind and number
1025  * of <code>FE_DGQ</code> elements. Note that the Raviart-Thomas element
1026  * already has <code>dim</code> vector components, so that the coupled
1027  * element will have <code>dim+1</code> vector components, the first
1028  * <code>dim</code> of which correspond to the velocity variable whereas the
1029  * last one corresponds to the pressure.
1030  *
1031 
1032  *
1033  * It is also worth comparing the way we constructed this element from its
1034  * base elements, with the way we have done so in @ref step_8 "step-8": there, we have
1035  * built it as <code>fe (FE_Q@<dim@>(1), dim)</code>, i.e. we have simply
1036  * used <code>dim</code> copies of the <code>FE_Q(1)</code> element, one
1037  * copy for the displacement in each coordinate direction.
1038  *
1039  * @code
1040  * template <int dim>
1041  * MixedLaplaceProblem<dim>::MixedLaplaceProblem(const unsigned int degree)
1042  * : degree(degree)
1043  * , fe(FE_RaviartThomas<dim>(degree), 1, FE_DGQ<dim>(degree), 1)
1044  * , dof_handler(triangulation)
1045  * {}
1046  *
1047  *
1048  *
1049  * @endcode

1052  * <a name="MixedLaplaceProblemmake_grid_and_dofs"></a>
1053  * <h4>MixedLaplaceProblem::make_grid_and_dofs</h4>
1054  *
1055 
1056  *
1057  * This next function starts out with well-known functions calls that create
1058  * and refine a mesh, and then associate degrees of freedom with it:
1059  *
1060  * @code
1061  * template <int dim>
1062  * void MixedLaplaceProblem<dim>::make_grid_and_dofs()
1063  * {
1064  * GridGenerator::hyper_cube(triangulation, -1, 1);
1065  * triangulation.refine_global(5);
1066  *
1067  * dof_handler.distribute_dofs(fe);
1068  *
1069  * @endcode
1070  *
1071  * However, then things become different. As mentioned in the
1072  * introduction, we want to subdivide the matrix into blocks corresponding
1073  * to the two different kinds of variables, velocity and pressure. To this
1074  * end, we first have to make sure that the indices corresponding to
1075  * velocities and pressures are not intermingled: First all velocity
1076  * degrees of freedom, then all pressure DoFs. This way, the global matrix
1077  * separates nicely into a @f$2 \times 2@f$ system. To achieve this, we have to
1078  * renumber degrees of freedom based on their vector component, an
1079  * operation that conveniently is already implemented:
1080  *
1081  * @code
1082  * DoFRenumbering::component_wise(dof_handler);
1083  *
1084  * @endcode
1085  *
1086  * The next thing is that we want to figure out the sizes of these blocks
1087  * so that we can allocate an appropriate amount of space. To this end, we
1088  * call the DoFTools::count_dofs_per_fe_component() function that
1089  * counts how many shape functions are non-zero for a particular vector
1090  * component. We have <code>dim+1</code> vector components, and
1091  * DoFTools::count_dofs_per_fe_component() will count how many shape
1092  * functions belong to each of these components.
1093  *
1094 
1095  *
1096  * There is one problem here. As described in the documentation of that
1097  * function, it <i>wants</i> to put the number of @f$x@f$-velocity shape
1098  * functions into <code>dofs_per_component[0]</code>, the number of
1099  * @f$y@f$-velocity shape functions into <code>dofs_per_component[1]</code>
1100  * (and similar in 3d), and the number of pressure shape functions into
1101  * <code>dofs_per_component[dim]</code>. But, the Raviart-Thomas element
1102  * is special in that it is non-@ref GlossPrimitive "primitive", i.e.,
1103  * for Raviart-Thomas elements all velocity shape functions
1104  * are nonzero in all components. In other words, the function cannot
1105  * distinguish between @f$x@f$ and @f$y@f$ velocity functions because there
1106  * <i>is</i> no such distinction. It therefore puts the overall number
1107  * of velocity into each of <code>dofs_per_component[c]</code>,
1108  * @f$0\le c\le \text{dim}@f$. On the other hand, the number
1109  * of pressure variables equals the number of shape functions that are
1110  * nonzero in the dim-th component.
1111  *
1112 
1113  *
1114  * Using this knowledge, we can get the number of velocity shape
1115  * functions from any of the first <code>dim</code> elements of
1116  * <code>dofs_per_component</code>, and then use this below to initialize
1117  * the vector and matrix block sizes, as well as create output.
1118  *
1119 
1120  *
1121  * @note If you find this concept difficult to understand, you may
1122  * want to consider using the function DoFTools::count_dofs_per_fe_block()
1123  * instead, as we do in the corresponding piece of code in @ref step_22 "step-22".
1124  * You might also want to read up on the difference between
1125  * @ref GlossBlock "blocks" and @ref GlossComponent "components"
1126  * in the glossary.
1127  *
1128  * @code
1129  * const std::vector<types::global_dof_index> dofs_per_component =
1130  * DoFTools::count_dofs_per_fe_component(dof_handler);
1131  * const unsigned int n_u = dofs_per_component[0],
1132  * n_p = dofs_per_component[dim];
1133  *
1134  * std::cout << "Number of active cells: " << triangulation.n_active_cells()
1135  * << std::endl
1136  * << "Total number of cells: " << triangulation.n_cells()
1137  * << std::endl
1138  * << "Number of degrees of freedom: " << dof_handler.n_dofs()
1139  * << " (" << n_u << '+' << n_p << ')' << std::endl;
1140  *
1141  * @endcode
1142  *
1143  * The next task is to allocate a sparsity pattern for the matrix that we
1144  * will create. We use a compressed sparsity pattern like in the previous
1145  * steps, but as <code>system_matrix</code> is a block matrix we use the
1146  * class <code>BlockDynamicSparsityPattern</code> instead of just
1147  * <code>DynamicSparsityPattern</code>. This block sparsity pattern has
1148  * four blocks in a @f$2 \times 2@f$ pattern. The blocks' sizes depend on
1149  * <code>n_u</code> and <code>n_p</code>, which hold the number of velocity
1150  * and pressure variables. In the second step we have to instruct the block
1151  * system to update its knowledge about the sizes of the blocks it manages;
1152  * this happens with the <code>dsp.collect_sizes ()</code> call.
1153  *
1154  * @code
1155  * BlockDynamicSparsityPattern dsp(2, 2);
1156  * dsp.block(0, 0).reinit(n_u, n_u);
1157  * dsp.block(1, 0).reinit(n_p, n_u);
1158  * dsp.block(0, 1).reinit(n_u, n_p);
1159  * dsp.block(1, 1).reinit(n_p, n_p);
1160  * dsp.collect_sizes();
1161  * DoFTools::make_sparsity_pattern(dof_handler, dsp);
1162  *
1163  * @endcode
1164  *
1165  * We use the compressed block sparsity pattern in the same way as the
1166  * non-block version to create the sparsity pattern and then the system
1167  * matrix:
1168  *
1169  * @code
1170  * sparsity_pattern.copy_from(dsp);
1171  * system_matrix.reinit(sparsity_pattern);
1172  *
1173  * @endcode
1174  *
1175  * Then we have to resize the solution and right hand side vectors in
1176  * exactly the same way as the block compressed sparsity pattern:
1177  *
1178  * @code
1179  * solution.reinit(2);
1180  * solution.block(0).reinit(n_u);
1181  * solution.block(1).reinit(n_p);
1182  * solution.collect_sizes();
1183  *
1184  * system_rhs.reinit(2);
1185  * system_rhs.block(0).reinit(n_u);
1186  * system_rhs.block(1).reinit(n_p);
1187  * system_rhs.collect_sizes();
1188  * }
1189  *
1190  *
1191  * @endcode
1192  *
1193  *
1194  * <a name="MixedLaplaceProblemassemble_system"></a>
1195  * <h4>MixedLaplaceProblem::assemble_system</h4>
1196  *
1197 
1198  *
1199  * Similarly, the function that assembles the linear system has mostly been
1200  * discussed already in the introduction to this example. At its top, what
1201  * happens are all the usual steps, with the addition that we do not only
1202  * allocate quadrature and <code>FEValues</code> objects for the cell terms,
1203  * but also for face terms. After that, we define the usual abbreviations
1204  * for variables, and the allocate space for the local matrix and right hand
1205  * side contributions, and the array that holds the global numbers of the
1206  * degrees of freedom local to the present cell.
1207  *
1208  * @code
1209  * template <int dim>
1210  * void MixedLaplaceProblem<dim>::assemble_system()
1211  * {
1212  * QGauss<dim> quadrature_formula(degree + 2);
1213  * QGauss<dim - 1> face_quadrature_formula(degree + 2);
1214  *
1215  * FEValues<dim> fe_values(fe,
1216  * quadrature_formula,
1217  * update_values | update_gradients |
1218  * update_quadrature_points | update_JxW_values);
1219  * FEFaceValues<dim> fe_face_values(fe,
1220  * face_quadrature_formula,
1221  * update_values | update_normal_vectors |
1222  * update_quadrature_points |
1223  * update_JxW_values);
1224  *
1225  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
1226  * const unsigned int n_q_points = quadrature_formula.size();
1227  * const unsigned int n_face_q_points = face_quadrature_formula.size();
1228  *
1229  * FullMatrix<double> local_matrix(dofs_per_cell, dofs_per_cell);
1230  * Vector<double> local_rhs(dofs_per_cell);
1231  *
1232  * std::vector<types::global_dof_index> local_dof_indices(dofs_per_cell);
1233  *
1234  * @endcode
1235  *
1236  * The next step is to declare objects that represent the source term,
1237  * pressure boundary value, and coefficient in the equation. In addition
1238  * to these objects that represent continuous functions, we also need
1239  * arrays to hold their values at the quadrature points of individual
1240  * cells (or faces, for the boundary values). Note that in the case of the
1241  * coefficient, the array has to be one of matrices.
1242  *
1243  * @code
1244  * const PrescribedSolution::RightHandSide<dim> right_hand_side;
1245  * const PrescribedSolution::PressureBoundaryValues<dim>
1246  * pressure_boundary_values;
1247  * const PrescribedSolution::KInverse<dim> k_inverse;
1248  *
1249  * std::vector<double> rhs_values(n_q_points);
1250  * std::vector<double> boundary_values(n_face_q_points);
1251  * std::vector<Tensor<2, dim>> k_inverse_values(n_q_points);
1252  *
1253  * @endcode
1254  *
1255  * Finally, we need a couple of extractors that we will use to get at the
1256  * velocity and pressure components of vector-valued shape
1257  * functions. Their function and use is described in detail in the @ref
1258  * vector_valued report. Essentially, we will use them as subscripts on
1259  * the FEValues objects below: the FEValues object describes all vector
1260  * components of shape functions, while after subscription, it will only
1261  * refer to the velocities (a set of <code>dim</code> components starting
1262  * at component zero) or the pressure (a scalar component located at
1263  * position <code>dim</code>):
1264  *
1265  * @code
1266  * const FEValuesExtractors::Vector velocities(0);
1267  * const FEValuesExtractors::Scalar pressure(dim);
1268  *
1269  * @endcode
1270  *
1271  * With all this in place, we can go on with the loop over all cells. The
1272  * body of this loop has been discussed in the introduction, and will not
1273  * be commented any further here:
1274  *
1275  * @code
1276  * for (const auto &cell : dof_handler.active_cell_iterators())
1277  * {
1278  * fe_values.reinit(cell);
1279  * local_matrix = 0;
1280  * local_rhs = 0;
1281  *
1282  * right_hand_side.value_list(fe_values.get_quadrature_points(),
1283  * rhs_values);
1284  * k_inverse.value_list(fe_values.get_quadrature_points(),
1285  * k_inverse_values);
1286  *
1287  * for (unsigned int q = 0; q < n_q_points; ++q)
1288  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1289  * {
1290  * const Tensor<1, dim> phi_i_u = fe_values[velocities].value(i, q);
1291  * const double div_phi_i_u = fe_values[velocities].divergence(i, q);
1292  * const double phi_i_p = fe_values[pressure].value(i, q);
1293  *
1294  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
1295  * {
1296  * const Tensor<1, dim> phi_j_u =
1297  * fe_values[velocities].value(j, q);
1298  * const double div_phi_j_u =
1299  * fe_values[velocities].divergence(j, q);
1300  * const double phi_j_p = fe_values[pressure].value(j, q);
1301  *
1302  * local_matrix(i, j) +=
1303  * (phi_i_u * k_inverse_values[q] * phi_j_u
1304  * - phi_i_p * div_phi_j_u
1305  * - div_phi_i_u * phi_j_p)
1306  * * fe_values.JxW(q);
1307  * }
1308  *
1309  * local_rhs(i) += -phi_i_p * rhs_values[q] * fe_values.JxW(q);
1310  * }
1311  *
1312  * for (const auto &face : cell->face_iterators())
1313  * if (face->at_boundary())
1314  * {
1315  * fe_face_values.reinit(cell, face);
1316  *
1317  * pressure_boundary_values.value_list(
1318  * fe_face_values.get_quadrature_points(), boundary_values);
1319  *
1320  * for (unsigned int q = 0; q < n_face_q_points; ++q)
1321  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1322  * local_rhs(i) += -(fe_face_values[velocities].value(i, q) *
1323  * fe_face_values.normal_vector(q) *
1324  * boundary_values[q] *
1325  * fe_face_values.JxW(q));
1326  * }
1327  *
1328  * @endcode
1329  *
1330  * The final step in the loop over all cells is to transfer local
1331  * contributions into the global matrix and right hand side
1332  * vector. Note that we use exactly the same interface as in previous
1333  * examples, although we now use block matrices and vectors instead of
1334  * the regular ones. In other words, to the outside world, block
1335  * objects have the same interface as matrices and vectors, but they
1336  * additionally allow to access individual blocks.
1337  *
1338  * @code
1339  * cell->get_dof_indices(local_dof_indices);
1340  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1341  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
1342  * system_matrix.add(local_dof_indices[i],
1343  * local_dof_indices[j],
1344  * local_matrix(i, j));
1345  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1346  * system_rhs(local_dof_indices[i]) += local_rhs(i);
1347  * }
1348  * }

1351  * @endcode

1359  * The linear solvers and preconditioners we use in this example have

1368  * <a name="MixedLaplacesolve"></a>
1369  * <h4>MixedLaplace::solve</h4>
1370  *
1371 
1372  *
1373  * As already outlined in the introduction, the solve function consists
1374  * essentially of two steps. First, we have to form the first equation
1375  * involving the Schur complement and solve for the pressure (component 1
1376  * of the solution). Then, we can reconstruct the velocities from the
1377  * second equation (component 0 of the solution).
1378  *
1379  * @code
1380  * template <int dim>
1381  * void MixedLaplaceProblem<dim>::solve()
1382  * {
1383  * @endcode
1384  *
1385  * As a first step we declare references to all block components of the
1386  * matrix, the right hand side and the solution vector that we will
1387  * need.
1388  *
1389  * @code
1390  * const auto &M = system_matrix.block(0, 0);
1391  * const auto &B = system_matrix.block(0, 1);
1392  *
1393  * const auto &F = system_rhs.block(0);
1394  * const auto &G = system_rhs.block(1);
1395  *
1396  * auto &U = solution.block(0);
1397  * auto &P = solution.block(1);
1398  *
1399  * @endcode
1400  *
1401  * Then, we will create corresponding LinearOperator objects and create
1402  * the <code>op_M_inv</code> operator:
1403  *
1404  * @code
1405  * const auto op_M = linear_operator(M);
1406  * const auto op_B = linear_operator(B);
1407  *
1408  * ReductionControl reduction_control_M(2000, 1.0e-18, 1.0e-10);
1409  * SolverCG<Vector<double>> solver_M(reduction_control_M);
1410  * PreconditionJacobi<SparseMatrix<double>> preconditioner_M;
1411  *
1412  * preconditioner_M.initialize(M);
1413  *
1414  * const auto op_M_inv = inverse_operator(op_M, solver_M, preconditioner_M);
1415  *
1416  * @endcode
1417  *
1418  * This allows us to declare the Schur complement <code>op_S</code> and
1419  * the approximate Schur complement <code>op_aS</code>:
1420  *
1421  * @code
1422  * const auto op_S = transpose_operator(op_B) * op_M_inv * op_B;
1423  * const auto op_aS =
1424  * transpose_operator(op_B) * linear_operator(preconditioner_M) * op_B;
1425  *
1426  * @endcode
1427  *
1428  * We now create a preconditioner out of <code>op_aS</code> that
1429  * applies a fixed number of 30 (inexpensive) CG iterations:
1430  *
1431  * @code
1432  * IterationNumberControl iteration_number_control_aS(30, 1.e-18);
1433  * SolverCG<Vector<double>> solver_aS(iteration_number_control_aS);
1434  *
1435  * const auto preconditioner_S =
1436  * inverse_operator(op_aS, solver_aS, PreconditionIdentity());
1437  *
1438  * @endcode
1439  *
1440  * Now on to the first equation. The right hand side of it is
1441  * @f$B^TM^{-1}F-G@f$, which is what we compute in the first few lines. We
1442  * then solve the first equation with a CG solver and the
1443  * preconditioner we just declared.
1444  *
1445  * @code
1446  * const auto schur_rhs = transpose_operator(op_B) * op_M_inv * F - G;
1447  *
1448  * SolverControl solver_control_S(2000, 1.e-12);
1449  * SolverCG<Vector<double>> solver_S(solver_control_S);
1450  *
1451  * const auto op_S_inv = inverse_operator(op_S, solver_S, preconditioner_S);
1452  *
1453  * P = op_S_inv * schur_rhs;
1454  *
1455  * std::cout << solver_control_S.last_step()
1456  * << " CG Schur complement iterations to obtain convergence."
1457  * << std::endl;
1458  *
1459  * @endcode
1460  *
1461  * After we have the pressure, we can compute the velocity. The equation
1462  * reads @f$MU=-BP+F@f$, and we solve it by first computing the right hand
1463  * side, and then multiplying it with the object that represents the
1464  * inverse of the mass matrix:
1465  *
1466  * @code
1467  * U = op_M_inv * (F - op_B * P);
1468  * }
1469  *
1470  *
1471  * @endcode

1480  * <a name="MixedLaplacecompute_errors"></a>
1481  * <h4>MixedLaplace::compute_errors</h4>
1482  *
1483 
1484  *
1485  * After we have dealt with the linear solver and preconditioners, we
1486  * continue with the implementation of our main class. In particular, the
1487  * next task is to compute the errors in our numerical solution, in both the
1488  * pressures as well as velocities.
1489  *
1490 
1491  *
1492  * To compute errors in the solution, we have already introduced the
1493  * <code>VectorTools::integrate_difference</code> function in @ref step_7 "step-7" and
1494  * @ref step_11 "step-11". However, there we only dealt with scalar solutions, whereas here
1495  * we have a vector-valued solution with components that even denote
1496  * different quantities and may have different orders of convergence (this
1497  * isn't the case here, by choice of the used finite elements, but is
1498  * frequently the case in mixed finite element applications). What we
1499  * therefore have to do is to `mask' the components that we are interested
1500  * in. This is easily done: the
1501  * <code>VectorTools::integrate_difference</code> function takes as one of its
1502  * arguments a pointer to a weight function (the parameter defaults to the
1503  * null pointer, meaning unit weights). What we have to do is to pass
1504  * a function object that equals one in the components we are interested in,
1505  * and zero in the other ones. For example, to compute the pressure error,
1506  * we should pass a function that represents the constant vector with a unit
1507  * value in component <code>dim</code>, whereas for the velocity the
1508  * constant vector should be one in the first <code>dim</code> components,
1509  * and zero in the location of the pressure.
1510  *
1511 
1512  *
1513  * In deal.II, the <code>ComponentSelectFunction</code> does exactly this:
1514  * it wants to know how many vector components the function it is to
1515  * represent should have (in our case this would be <code>dim+1</code>, for
1516  * the joint velocity-pressure space) and which individual or range of
1517  * components should be equal to one. We therefore define two such masks at
1518  * the beginning of the function, following by an object representing the
1519  * exact solution and a vector in which we will store the cellwise errors as
1520  * computed by <code>integrate_difference</code>:
1521  *
1522  * @code
1523  * template <int dim>
1524  * void MixedLaplaceProblem<dim>::compute_errors() const
1525  * {
1526  * const ComponentSelectFunction<dim> pressure_mask(dim, dim + 1);
1527  * const ComponentSelectFunction<dim> velocity_mask(std::make_pair(0, dim),
1528  * dim + 1);
1529  *
1530  * PrescribedSolution::ExactSolution<dim> exact_solution;
1531  * Vector<double> cellwise_errors(triangulation.n_active_cells());
1532  *
1533  * @endcode
1534  *
1535  * As already discussed in @ref step_7 "step-7", we have to realize that it is
1536  * impossible to integrate the errors exactly. All we can do is
1537  * approximate this integral using quadrature. This actually presents a
1538  * slight twist here: if we naively chose an object of type
1539  * <code>QGauss@<dim@>(degree+1)</code> as one may be inclined to do (this
1540  * is what we used for integrating the linear system), one realizes that
1541  * the error is very small and does not follow the expected convergence
1542  * curves at all. What is happening is that for the mixed finite elements
1543  * used here, the Gauss points happen to be superconvergence points in
1544  * which the pointwise error is much smaller (and converges with higher
1545  * order) than anywhere else. These are therefore not particularly good
1546  * points for integration. To avoid this problem, we simply use a
1547  * trapezoidal rule and iterate it <code>degree+2</code> times in each
1548  * coordinate direction (again as explained in @ref step_7 "step-7"):
1549  *
1550  * @code
1551  * QTrapezoid<1> q_trapez;
1552  * QIterated<dim> quadrature(q_trapez, degree + 2);
1553  *
1554  * @endcode
1555  *
1556  * With this, we can then let the library compute the errors and output
1557  * them to the screen:
1558  *
1559  * @code
1560  * VectorTools::integrate_difference(dof_handler,
1561  * solution,
1562  * exact_solution,
1563  * cellwise_errors,
1564  * quadrature,
1565  * VectorTools::L2_norm,
1566  * &pressure_mask);
1567  * const double p_l2_error =
1568  * VectorTools::compute_global_error(triangulation,
1569  * cellwise_errors,
1570  * VectorTools::L2_norm);
1571  *
1572  * VectorTools::integrate_difference(dof_handler,
1573  * solution,
1574  * exact_solution,
1575  * cellwise_errors,
1576  * quadrature,
1577  * VectorTools::L2_norm,
1578  * &velocity_mask);
1579  * const double u_l2_error =
1580  * VectorTools::compute_global_error(triangulation,
1581  * cellwise_errors,
1582  * VectorTools::L2_norm);
1583  *
1584  * std::cout << "Errors: ||e_p||_L2 = " << p_l2_error
1585  * << ", ||e_u||_L2 = " << u_l2_error << std::endl;
1586  * }
1587  *
1588  *
1589  * @endcode

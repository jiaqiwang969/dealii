388  *

390  * functions above. As before, we will write everything as templates that have
391  * a formal parameter <code>dim</code> that we assume unknown at the time we
392  * define the template functions. Only later, the compiler will find a
393  * declaration of <code>Step4@<2@></code> (in the <code>main</code> function,

395  * a process referred to as `instantiation of a template'. When doing so, it
396  * will also replace instances of <code>RightHandSide@<dim@></code> by
397  * <code>RightHandSide@<2@></code> and instantiate the latter class from the

443  *
444  * Grid creation is something inherently dimension dependent. However, as long

446  * abstract for you. In our case, we would like to again solve on the square

504  * <a name="Step4assemble_system"></a>
505  * <h4>Step4::assemble_system</h4>
506  *
507 
508  *
509  * Unlike in the previous example, we would now like to use a non-constant
510  * right hand side function and non-zero boundary values. Both are tasks that
511  * are readily achieved with only a few new lines of code in the assemblage of
512  * the matrix and right hand side.
513  *
514 
515  *
516  * More interesting, though, is the way we assemble matrix and right hand side
517  * vector dimension independently: there is simply no difference to the
518  * two-dimensional case. Since the important objects used in this function
519  * (quadrature formula, FEValues) depend on the dimension by way of a template
520  * parameter as well, they can take care of setting up properly everything for
521  * the dimension for which this function is compiled. By declaring all classes
522  * which might depend on the dimension using a template parameter, the library
523  * can make nearly all work for you and you don't have to care about most
524  * things.
525  *
526  * @code
527  * template <int dim>
528  * void Step4<dim>::assemble_system()
529  * {
530  * QGauss<dim> quadrature_formula(fe.degree + 1);
531  *
532  * @endcode
533  *
534  * We wanted to have a non-constant right hand side, so we use an object of
535  * the class declared above to generate the necessary data. Since this right
536  * hand side object is only used locally in the present function, we declare
537  * it here as a local variable:
538  *
539  * @code
540  * RightHandSide<dim> right_hand_side;
541  *
542  * @endcode
543  *
544  * Compared to the previous example, in order to evaluate the non-constant
545  * right hand side function we now also need the quadrature points on the
546  * cell we are presently on (previously, we only required values and
547  * gradients of the shape function from the FEValues object, as well as the
548  * quadrature weights, FEValues::JxW() ). We can tell the FEValues object to
549  * do for us by also giving it the #update_quadrature_points flag:
550  *
551  * @code
552  * FEValues<dim> fe_values(fe,
553  * quadrature_formula,
554  * update_values | update_gradients |
555  * update_quadrature_points | update_JxW_values);
556  *
557  * @endcode
558  *
559  * We then again define the same abbreviation as in the previous program.
560  * The value of this variable of course depends on the dimension which we
561  * are presently using, but the FiniteElement class does all the necessary
562  * work for you and you don't have to care about the dimension dependent
563  * parts:
564  *
565  * @code
566  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
567  *
568  * FullMatrix<double> cell_matrix(dofs_per_cell, dofs_per_cell);
569  * Vector<double> cell_rhs(dofs_per_cell);
570  *
571  * std::vector<types::global_dof_index> local_dof_indices(dofs_per_cell);
572  *
573  * @endcode
574  *
575  * Next, we again have to loop over all cells and assemble local
576  * contributions. Note, that a cell is a quadrilateral in two space
577  * dimensions, but a hexahedron in 3D. In fact, the
578  * <code>active_cell_iterator</code> data type is something different,
579  * depending on the dimension we are in, but to the outside world they look
580  * alike and you will probably never see a difference. In any case, the real
581  * type is hidden by using `auto`:
582  *
583  * @code
584  * for (const auto &cell : dof_handler.active_cell_iterators())
585  * {
586  * fe_values.reinit(cell);
587  * cell_matrix = 0;
588  * cell_rhs = 0;
589  *
590  * @endcode
591  *
592  * Now we have to assemble the local matrix and right hand side. This is
593  * done exactly like in the previous example, but now we revert the
594  * order of the loops (which we can safely do since they are independent
595  * of each other) and merge the loops for the local matrix and the local
596  * vector as far as possible to make things a bit faster.
597  *
598 
599  *
600  * Assembling the right hand side presents the only significant
601  * difference to how we did things in @ref step_3 "step-3": Instead of using a
602  * constant right hand side with value 1, we use the object representing
603  * the right hand side and evaluate it at the quadrature points:
604  *
605  * @code
606  * for (const unsigned int q_index : fe_values.quadrature_point_indices())
607  * for (const unsigned int i : fe_values.dof_indices())
608  * {
609  * for (const unsigned int j : fe_values.dof_indices())
610  * cell_matrix(i, j) +=
611  * (fe_values.shape_grad(i, q_index) * // grad phi_i(x_q)
612  * fe_values.shape_grad(j, q_index) * // grad phi_j(x_q)
613  * fe_values.JxW(q_index)); // dx
614  *
615  * const auto &x_q = fe_values.quadrature_point(q_index);
616  * cell_rhs(i) += (fe_values.shape_value(i, q_index) * // phi_i(x_q)
617  * right_hand_side.value(x_q) * // f(x_q)
618  * fe_values.JxW(q_index)); // dx
619  * }
620  * @endcode
621  *
622  * As a final remark to these loops: when we assemble the local
623  * contributions into <code>cell_matrix(i,j)</code>, we have to multiply
624  * the gradients of shape functions @f$i@f$ and @f$j@f$ at point number
625  * q_index and
626  * multiply it with the scalar weights JxW. This is what actually
627  * happens: <code>fe_values.shape_grad(i,q_index)</code> returns a
628  * <code>dim</code> dimensional vector, represented by a
629  * <code>Tensor@<1,dim@></code> object, and the operator* that
630  * multiplies it with the result of
631  * <code>fe_values.shape_grad(j,q_index)</code> makes sure that the
632  * <code>dim</code> components of the two vectors are properly
633  * contracted, and the result is a scalar floating point number that
634  * then is multiplied with the weights. Internally, this operator* makes
635  * sure that this happens correctly for all <code>dim</code> components
636  * of the vectors, whether <code>dim</code> be 2, 3, or any other space
637  * dimension; from a user's perspective, this is not something worth
638  * bothering with, however, making things a lot simpler if one wants to
639  * write code dimension independently.
640  *
641 
642  *
643  * With the local systems assembled, the transfer into the global matrix
644  * and right hand side is done exactly as before, but here we have again
645  * merged some loops for efficiency:
646  *
647  * @code
648  * cell->get_dof_indices(local_dof_indices);
649  * for (const unsigned int i : fe_values.dof_indices())
650  * {
651  * for (const unsigned int j : fe_values.dof_indices())
652  * system_matrix.add(local_dof_indices[i],
653  * local_dof_indices[j],
654  * cell_matrix(i, j));
655  *
656  * system_rhs(local_dof_indices[i]) += cell_rhs(i);
657  * }
658  * }
659  *
660  * @endcode
661  *
662  * As the final step in this function, we wanted to have non-homogeneous
663  * boundary values in this example, unlike the one before. This is a simple
664  * task, we only have to replace the Functions::ZeroFunction used there by an
665  * object of the class which describes the boundary values we would like to
666  * use (i.e. the <code>BoundaryValues</code> class declared above):
667  *
668 
669  *
670  * The function VectorTools::interpolate_boundary_values() will only work
671  * on faces that have been marked with boundary indicator 0 (because that's
672  * what we say the function should work on with the second argument below).
673  * If there are faces with boundary id other than 0, then the function
674  * interpolate_boundary_values will do nothing on these faces. For
675  * the Laplace equation doing nothing is equivalent to assuming that
676  * on those parts of the boundary a zero Neumann boundary condition holds.
677  *
678  * @code
679  * std::map<types::global_dof_index, double> boundary_values;
680  * VectorTools::interpolate_boundary_values(dof_handler,
681  * 0,
682  * BoundaryValues<dim>(),
683  * boundary_values);
684  * MatrixTools::apply_boundary_values(boundary_values,
685  * system_matrix,
686  * solution,
687  * system_rhs);
688  * }
689  *
690  *
691  * @endcode
692  *
693  *
694  * <a name="Step4solve"></a>
695  * <h4>Step4::solve</h4>

698  *
699  * Solving the linear system of equations is something that looks almost

702  *
703  * @code
704  * template <int dim>
705  * void Step4<dim>::solve()
706  * {
707  * SolverControl solver_control(1000, 1e-12);
708  * SolverCG<Vector<double>> solver(solver_control);
709  * solver.solve(system_matrix, solution, system_rhs, PreconditionIdentity());
710  *
711  * @endcode
712  *
713  * We have made one addition, though: since we suppress output from the

716  * @code

788  * <a name="Thecodemaincodefunction"></a>
789  * <h3>The <code>main</code> function</h3>
790  *
791 
792  *
793  * And this is the main function. It also looks mostly like in @ref step_3 "step-3", but if
794  * you look at the code below, note how we first create a variable of type
795  * <code>Step4@<2@></code> (forcing the compiler to compile the class template
796  * with <code>dim</code> replaced by <code>2</code>) and run a 2d simulation,
797  * and then we do the whole thing over in 3d.
798  *
799 
800  *
801  * In practice, this is probably not what you would do very frequently (you
802  * probably either want to solve a 2d problem, or one in 3d, but not both at
803  * the same time). However, it demonstrates the mechanism by which we can
804  * simply change which dimension we want in a single place, and thereby force
805  * the compiler to recompile the dimension independent class templates for the
806  * dimension we request. The emphasis here lies on the fact that we only need
807  * to change a single place. This makes it rather trivial to debug the program
808  * in 2d where computations are fast, and then switch a single place to a 3 to
809  * run the much more computing intensive program in 3d for `real'
810  * computations.
811  *
812 
813  *
814  * Each of the two blocks is enclosed in braces to make sure that the
815  * <code>laplace_problem_2d</code> variable goes out of scope (and releases
816  * the memory it holds) before we move on to allocate memory for the 3d
817  * case. Without the additional braces, the <code>laplace_problem_2d</code>
818  * variable would only be destroyed at the end of the function, i.e. after
819  * running the 3d problem, and would needlessly hog memory while the 3d run
820  * could actually use it.
821  *
822  * @code
823  * int main()
824  * {
825  * {
826  * Step4<2> laplace_problem_2d;
827  * laplace_problem_2d.run();
828  * }
829  *
830  * {
831  * Step4<3> laplace_problem_3d;
832  * laplace_problem_3d.run();
833  * }
834  *
835  * return 0;
836  * }
837  * @endcode

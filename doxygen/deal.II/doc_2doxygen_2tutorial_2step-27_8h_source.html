750  * <a name="LaplaceProblemLaplaceProblemconstructor"></a>
751  * <h4>LaplaceProblem::LaplaceProblem constructor</h4>
752  *
753 
754  *
755  * The constructor of this class is fairly straightforward. It associates
756  * the DoFHandler object with the triangulation, and then sets the
757  * maximal polynomial degree to 7 (in 1d and 2d) or 5 (in 3d and higher). We
758  * do so because using higher order polynomial degrees becomes prohibitively
759  * expensive, especially in higher space dimensions.
760  *
761 
762  *
763  * Following this, we fill the collections of finite element, and cell and
764  * face quadrature objects. We start with quadratic elements, and each
765  * quadrature formula is chosen so that it is appropriate for the matching
766  * finite element in the hp::FECollection object.
767  *
768  * @code
769  * template <int dim>
770  * LaplaceProblem<dim>::LaplaceProblem()
771  * : dof_handler(triangulation)
772  * , max_degree(dim <= 2 ? 7 : 5)
773  * {
774  * for (unsigned int degree = 2; degree <= max_degree; ++degree)
775  * {
776  * fe_collection.push_back(FE_Q<dim>(degree));
777  * quadrature_collection.push_back(QGauss<dim>(degree + 1));
778  * face_quadrature_collection.push_back(QGauss<dim - 1>(degree + 1));
779  * }
780  * }

843  * <a name="LaplaceProblemassemble_system"></a>
844  * <h4>LaplaceProblem::assemble_system</h4>
845  *
846 
847  *
848  * This is the function that assembles the global matrix and right hand side
849  * vector from the local contributions of each cell. Its main working is as
850  * has been described in many of the tutorial programs before. The
851  * significant deviations are the ones necessary for <i>hp</i> finite
852  * element methods. In particular, that we need to use a collection of
853  * FEValues object (implemented through the hp::FEValues class), and that we
854  * have to eliminate constrained degrees of freedom already when copying
855  * local contributions into global objects. Both of these are explained in
856  * detail in the introduction of this program.
857  *
858 
859  *
860  * One other slight complication is the fact that because we use different
861  * polynomial degrees on different cells, the matrices and vectors holding
862  * local contributions do not have the same size on all cells. At the
863  * beginning of the loop over all cells, we therefore each time have to
864  * resize them to the correct size (given by <code>dofs_per_cell</code>).
865  * Because these classes are implemented in such a way that reducing the size
866  * of a matrix or vector does not release the currently allocated memory
867  * (unless the new size is zero), the process of resizing at the beginning of
868  * the loop will only require re-allocation of memory during the first few
869  * iterations. Once we have found in a cell with the maximal finite element
870  * degree, no more re-allocations will happen because all subsequent
871  * <code>reinit</code> calls will only set the size to something that fits the
872  * currently allocated memory. This is important since allocating memory is
873  * expensive, and doing so every time we visit a new cell would take
874  * significant compute time.
875  *
876  * @code
877  * template <int dim>
878  * void LaplaceProblem<dim>::assemble_system()
879  * {
880  * hp::FEValues<dim> hp_fe_values(fe_collection,
881  * quadrature_collection,
882  * update_values | update_gradients |
883  * update_quadrature_points |
884  * update_JxW_values);
885  *
886  * RightHandSide<dim> rhs_function;
887  *
888  * FullMatrix<double> cell_matrix;
889  * Vector<double> cell_rhs;
890  *
891  * std::vector<types::global_dof_index> local_dof_indices;
892  *
893  * for (const auto &cell : dof_handler.active_cell_iterators())
894  * {
895  * const unsigned int dofs_per_cell = cell->get_fe().n_dofs_per_cell();
896  *
897  * cell_matrix.reinit(dofs_per_cell, dofs_per_cell);
898  * cell_matrix = 0;
899  *
900  * cell_rhs.reinit(dofs_per_cell);
901  * cell_rhs = 0;
902  *
903  * hp_fe_values.reinit(cell);
904  *
905  * const FEValues<dim> &fe_values = hp_fe_values.get_present_fe_values();
906  *
907  * std::vector<double> rhs_values(fe_values.n_quadrature_points);
908  * rhs_function.value_list(fe_values.get_quadrature_points(), rhs_values);
909  *
910  * for (unsigned int q_point = 0; q_point < fe_values.n_quadrature_points;
911  * ++q_point)
912  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
913  * {
914  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
915  * cell_matrix(i, j) +=
916  * (fe_values.shape_grad(i, q_point) * // grad phi_i(x_q)
917  * fe_values.shape_grad(j, q_point) * // grad phi_j(x_q)
918  * fe_values.JxW(q_point)); // dx
919  *
920  * cell_rhs(i) += (fe_values.shape_value(i, q_point) * // phi_i(x_q)
921  * rhs_values[q_point] * // f(x_q)
922  * fe_values.JxW(q_point)); // dx
923  * }
924  *
925  * local_dof_indices.resize(dofs_per_cell);
926  * cell->get_dof_indices(local_dof_indices);
927  *
928  * constraints.distribute_local_to_global(
929  * cell_matrix, cell_rhs, local_dof_indices, system_matrix, system_rhs);
930  * }
931  * }
932  *
933  *
934  *
935  * @endcode

968  * <a name="LaplaceProblempostprocess"></a>
969  * <h4>LaplaceProblem::postprocess</h4>
970  *
971 
972  *
973  * After solving the linear system, we will want to postprocess the
974  * solution. Here, all we do is to estimate the error, estimate the local
975  * smoothness of the solution as described in the introduction, then write
976  * graphical output, and finally refine the mesh in both @f$h@f$ and @f$p@f$
977  * according to the indicators computed before. We do all this in the same
978  * function because we want the estimated error and smoothness indicators
979  * not only for refinement, but also include them in the graphical output.
980  *
981  * @code
982  * template <int dim>
983  * void LaplaceProblem<dim>::postprocess(const unsigned int cycle)
984  * {
985  * @endcode
986  *
987  * Let us start with computing estimated error and smoothness indicators,
988  * which each are one number for each active cell of our
989  * triangulation. For the error indicator, we use the KellyErrorEstimator
990  * class as always.
991  *
992  * @code
993  * Vector<float> estimated_error_per_cell(triangulation.n_active_cells());
994  * KellyErrorEstimator<dim>::estimate(
995  * dof_handler,
996  * face_quadrature_collection,
997  * std::map<types::boundary_id, const Function<dim> *>(),
998  * solution,
999  * estimated_error_per_cell);
1000  *
1001  * @endcode
1002  *
1003  * Estimating the smoothness is performed with the method of decaying
1004  * expansion coefficients as outlined in the introduction. We will first
1005  * need to create an object capable of transforming the finite element
1006  * solution on every single cell into a sequence of Fourier series
1007  * coefficients. The SmoothnessEstimator namespace offers a factory function
1008  * for such a FESeries::Fourier object that is optimized for the process of
1009  * estimating smoothness. The actual determination of the decay of Fourier
1010  * coefficients on every individual cell then happens in the last function.
1011  *
1012  * @code
1013  * Vector<float> smoothness_indicators(triangulation.n_active_cells());
1014  * FESeries::Fourier<dim> fourier =
1015  * SmoothnessEstimator::Fourier::default_fe_series(fe_collection);
1016  * SmoothnessEstimator::Fourier::coefficient_decay(fourier,
1017  * dof_handler,
1018  * solution,
1019  * smoothness_indicators);
1020  *
1021  * @endcode
1022  *
1023  * Next we want to generate graphical output. In addition to the two
1024  * estimated quantities derived above, we would also like to output the
1025  * polynomial degree of the finite elements used on each of the elements
1026  * on the mesh.
1027  *
1028 
1029  *
1030  * The way to do that requires that we loop over all cells and poll the
1031  * active finite element index of them using
1032  * <code>cell-@>active_fe_index()</code>. We then use the result of this
1033  * operation and query the finite element collection for the finite
1034  * element with that index, and finally determine the polynomial degree of
1035  * that element. The result we put into a vector with one element per
1036  * cell. The DataOut class requires this to be a vector of
1037  * <code>float</code> or <code>double</code>, even though our values are
1038  * all integers, so that is what we use:
1039  *
1040  * @code
1041  * {
1042  * Vector<float> fe_degrees(triangulation.n_active_cells());
1043  * for (const auto &cell : dof_handler.active_cell_iterators())
1044  * fe_degrees(cell->active_cell_index()) =
1045  * fe_collection[cell->active_fe_index()].degree;
1046  *
1047  * @endcode
1048  *
1049  * With now all data vectors available -- solution, estimated errors and
1050  * smoothness indicators, and finite element degrees --, we create a
1051  * DataOut object for graphical output and attach all data:
1052  *
1053  * @code
1054  * DataOut<dim> data_out;
1055  *
1056  * data_out.attach_dof_handler(dof_handler);
1057  * data_out.add_data_vector(solution, "solution");
1058  * data_out.add_data_vector(estimated_error_per_cell, "error");
1059  * data_out.add_data_vector(smoothness_indicators, "smoothness");
1060  * data_out.add_data_vector(fe_degrees, "fe_degree");
1061  * data_out.build_patches();
1062  *
1063  * @endcode
1064  *
1065  * The final step in generating output is to determine a file name, open
1066  * the file, and write the data into it (here, we use VTK format):
1067  *
1068  * @code
1069  * const std::string filename =
1070  * "solution-" + Utilities::int_to_string(cycle, 2) + ".vtk";
1071  * std::ofstream output(filename);
1072  * data_out.write_vtk(output);
1073  * }
1074  *
1075  * @endcode
1076  *
1077  * After this, we would like to actually refine the mesh, in both @f$h@f$ and
1078  * @f$p@f$. The way we are going to do this is as follows: first, we use the
1079  * estimated error to flag those cells for refinement that have the
1080  * largest error. This is what we have always done:
1081  *
1082  * @code
1083  * {
1084  * GridRefinement::refine_and_coarsen_fixed_number(triangulation,
1085  * estimated_error_per_cell,
1086  * 0.3,
1087  * 0.03);
1088  *
1089  * @endcode
1090  *
1091  * Next we would like to figure out which of the cells that have been
1092  * flagged for refinement should actually have @f$p@f$ increased instead of
1093  * @f$h@f$ decreased. The strategy we choose here is that we look at the
1094  * smoothness indicators of those cells that are flagged for refinement,
1095  * and increase @f$p@f$ for those with a smoothness larger than a certain
1096  * relative threshold. In other words, for every cell for which (i) the
1097  * refinement flag is set, (ii) the smoothness indicator is larger than
1098  * the threshold, and (iii) we still have a finite element with a
1099  * polynomial degree higher than the current one in the finite element
1100  * collection, we will assign a future FE index that corresponds to a
1101  * polynomial with degree one higher than it currently is. The following
1102  * function is capable of doing exactly this. Absent any better
1103  * strategies, we will set the threshold via interpolation between the
1104  * minimal and maximal smoothness indicators on cells flagged for
1105  * refinement. Since the corner singularities are strongly localized, we
1106  * will favor @f$p@f$- over @f$h@f$-refinement quantitatively. We achieve this
1107  * with a low threshold by setting a small interpolation factor of 0.2. In
1108  * the same way, we deal with cells that are going to be coarsened and
1109  * decrease their polynomial degree when their smoothness indicator is
1110  * below the corresponding threshold determined on cells to be coarsened.
1111  *
1112  * @code
1113  * hp::Refinement::p_adaptivity_from_relative_threshold(
1114  * dof_handler, smoothness_indicators, 0.2, 0.2);
1115  *
1116  * @endcode
1117  *
1118  * The above function only determines whether the polynomial degree will
1119  * change via future FE indices, but does not manipulate the
1120  * @f$h@f$-refinement flags. So for cells that are flagged for both refinement
1121  * categories, we prefer @f$p@f$- over @f$h@f$-refinement. The following function
1122  * call ensures that only one of @f$p@f$- or @f$h@f$-refinement is imposed, and
1123  * not both at once.
1124  *
1125  * @code
1126  * hp::Refinement::choose_p_over_h(dof_handler);
1127  *
1128  * @endcode
1129  *
1130  * For grid adaptive refinement, we ensure a 2:1 mesh balance by limiting
1131  * the difference of refinement levels of neighboring cells to one by
1132  * calling Triangulation::prepare_coarsening_and_refinement(). We would
1133  * like to achieve something similar for the p-levels of neighboring
1134  * cells: levels of future finite elements are not allowed to differ by
1135  * more than a specified difference. With its default parameters, a call
1136  * of hp::Refinement::limit_p_level_difference() ensures that their level
1137  * difference is limited to one. This will not necessarily decrease the
1138  * number of hanging nodes in the domain, but makes sure that high order
1139  * polynomials are not constrained to much lower polynomials on faces,
1140  * e.g., fifth order to second order polynomials.
1141  *
1142  * @code
1143  * triangulation.prepare_coarsening_and_refinement();
1144  * hp::Refinement::limit_p_level_difference(dof_handler);
1145  *
1146  * @endcode
1147  *
1148  * At the end of this procedure, we then refine the mesh. During this
1149  * process, children of cells undergoing bisection inherit their mother
1150  * cell's finite element index. Further, future finite element indices
1151  * will turn into active ones, so that the new finite elements will be
1152  * assigned to cells after the next call of DoFHandler::distribute_dofs().
1153  *
1154  * @code
1155  * triangulation.execute_coarsening_and_refinement();
1156  * }
1157  * }
1158  *
1159  *
1160  * @endcode

1163  * <a name="LaplaceProblemcreate_coarse_grid"></a>
1164  * <h4>LaplaceProblem::create_coarse_grid</h4>
1165  *
1166 
1167  *
1168  * The following function is used when creating the initial grid. The grid we
1169  * would like to create is actually similar to the one from @ref step_14 "step-14", i.e., the
1170  * square domain with the square hole in the middle. It can be generated by
1171  * exactly the same function. However, since its implementation is only a
1172  * specialization of the 2d case, we will present a different way of creating
1173  * this domain which is dimension independent.
1174  *
1175

1177  * We first create a hypercube triangulation with enough cells so that it
1178  * already holds our desired domain @f$[-1,1]^d@f$, subdivided into @f$4^d@f$ cells.
1179  * We then remove those cells in the center of the domain by testing the
1180  * coordinate values of the vertices on each cell. In the end, we refine the

1183  * @code

1368 从这里可以得出的主要结论是，内角的规则性损失是一个高度局部的现象；它似乎只影响到与角本身相邻的单元，所以当我们细化网格时，黑色的颜色不再可见。除了角部，这一连串的图示意味着平滑度估计在某种程度上与网格细化无关，特别是当我们远离边界时。同样明显的是，平滑度估计值与解的实际尺寸无关（见上面的解的图片），这也是应该的。然而，更值得关注的一点是，人们在仔细观察后发现，我们的估计器高估了有悬空节点的单元格的解决方案的平滑度。这反过来又导致了这些区域的多项式度数更高，使有限元在单元上的分配出现偏差。
1369 
1370 对于这种效果，我们目前还没有很好的解释。一种理论是，在有悬挂节点的单元上的数值解当然是受限制的，因此不能完全自由地探索函数空间以接近精确解。这种自由度的缺乏可能表现为在这些单元上产生具有抑制振荡的数值解，意味着更高的平滑度。估计器会捕捉到这个信号，估计的平滑度会高估实际值。然而，这个程序的作者目前还没有得到关于发生了什么的明确答案。

1372 当然，更大的问题是如何避免这个问题。可能的做法包括不是在单个细胞上估计平滑度，而是在每个细胞周围的细胞集合体或斑块上估计平滑度。也可能为每个细胞找到简单的校正因子，这取决于它所拥有的受限自由度的数量。无论哪种情况，都有大量的机会来进一步研究寻找好的 @f$hp@f$ -精化标准。另一方面，目前方案的主要内容是在deal.II中演示使用 @f$hp@f$ -技术，这不受我们使用可能的次优细化标准的影响。

1382 <a name="Differenthpdecisionstrategies"></a><h4>Different hp-decision strategies</h4>
1383 
1384 
1385 本教程只演示了决定 @f$h@f$ -和 @f$p@f$ -适应的一个特殊策略。事实上，还有许多自动决定适应类型的方法，其中一些已经在deal.II中实现： <ul> <li> <i>Fourier coefficient decay:</i> 这是本教程中目前实现的策略。有关该策略的更多信息，请参见 SmoothnessEstimator::Fourier 命名空间的一般文档。 </li>
1386 
1387  <li> <i>Legendre coefficient decay:</i> 这个策略与目前的策略很相似，但使用了Legendre级数扩展，而不是傅里叶级数：这个策略使用Legendre多项式，而不是正弦波作为基础函数。当然，由于我们在每个单元上使用有限维度的多项式来近似解，因此解在Legendre多项式中的扩展也是有限的，因此，当我们谈论这个扩展的 "衰减 "时，我们只能考虑这个扩展的有限多个非零系数，而不是用渐进的方式来思考。 但是，如果我们有足够多的这些系数，我们当然可以把这些系数的衰减看作是精确解的系数衰减的特征（一般来说，精确解不是多项式的，所以会有一个无限的Legendre扩展），考虑我们拥有的系数应该可以揭示一些关于精确解的特性。
1388 
1389  从傅里叶策略过渡到Legendre策略是非常简单的。 你只需要改变序列扩展类和相应的平滑度估计函数，使其成为适当命名空间 FESeries::Legendre 和 SmoothnessEstimator::Legendre. 的一部分，这个策略在@ref step_75 "step-75"中使用。关于这个策略的理论背景，请参考 SmoothnessEstimator::Legendre 命名空间的一般文档，以及 @cite mavriplis1994hp 、 @cite eibner2007hp 和 @cite davydov2017hp 。 </li>
1390 
1391  <li> <i>Refinement history:</i> 最后一种策略与其他两种截然不同。在理论上，我们知道在改变函数空间离散化后，误差将如何收敛。用 @f$h@f$ -细化，解会以代数方式收敛，正如在步骤7中已经指出的。不过，如果解足够平滑，我们预计解将随着有限元的多项式程度的增加而指数级收敛。我们可以在下面的步骤中把对误差的正确预测与实际误差进行比较，看看我们对适应类型的选择是否合理。
1392 
1393  向这一策略的过渡要复杂一些。为此，我们需要一个纯 @f$h@f$ -或 @f$p@f$ -细化的初始化步骤，我们需要将预测的误差转移到适应的网格上。 hp::Refinement::predict_error() 函数的大量文档不仅描述了这种方法的理论细节，而且还介绍了如何在你的代码中实现这种策略的蓝图。欲了解更多信息，请参见 @cite melenk2001hp 。
1394 
1395  请注意，用这个特殊的函数，你无法预测时间依赖性问题的下一个时间步骤的误差。因此，这种策略不能不加思索地应用于这种类型的问题。另外，也可以使用下面的方法，这也适用于所有其他的策略：从每个时间步长的粗网格开始，不断细化，直到对结果满意，然后才进入下一个时间步长。 </li> </ul> 。
1396 
1397 试着在本教程中实施这些策略之一，观察结果的微妙变化。你会注意到，所有的策略都能够识别重心角附近的奇点，并在这些区域执行 @f$h@f$ -精简，而在体域中更倾向于 @f$p@f$ -精简。这些策略的详细比较见 @cite fehling2020 。

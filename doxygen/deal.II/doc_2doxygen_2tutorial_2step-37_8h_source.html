553  * <a name="Matrixfreeimplementation"></a>
554  * <h3>Matrix-free implementation</h3>
555  *
556 
557  *
558  * The following class, called <code>LaplaceOperator</code>, implements the
559  * differential operator. For all practical purposes, it is a matrix, i.e.,
560  * you can ask it for its size (member functions <code>m(), n()</code>) and
561  * you can apply it to a vector (the <code>vmult()</code> function). The
562  * difference to a real matrix of course lies in the fact that this class
563  * does not actually store the <i>elements</i> of the matrix, but only knows
564  * how to compute the action of the operator when applied to a vector.
565  *
566 
567  *
568  * The infrastructure describing the matrix size, the initialization from a
569  * MatrixFree object, and the various interfaces to matrix-vector products
570  * through vmult() and Tvmult() methods, is provided by the class
571  * MatrixFreeOperator::Base from which this class derives. The
572  * LaplaceOperator class defined here only has to provide a few interfaces,
573  * namely the actual action of the operator through the apply_add() method
574  * that gets used in the vmult() functions, and a method to compute the
575  * diagonal entries of the underlying matrix. We need the diagonal for the
576  * definition of the multigrid smoother. Since we consider a problem with
577  * variable coefficient, we further implement a method that can fill the
578  * coefficient values.
579  *
580 
581  *
582  * Note that the file <code>include/deal.II/matrix_free/operators.h</code>
583  * already contains an implementation of the Laplacian through the class
584  * MatrixFreeOperators::LaplaceOperator. For educational purposes, the
585  * operator is re-implemented in this tutorial program, explaining the
586  * ingredients and concepts used there.
587  *
588 
589  *
590  * This program makes use of the data cache for finite element operator
591  * application that is integrated in deal.II. This data cache class is
592  * called MatrixFree. It contains mapping information (Jacobians) and index
593  * relations between local and global degrees of freedom. It also contains
594  * constraints like the ones from hanging nodes or Dirichlet boundary
595  * conditions. Moreover, it can issue a loop over all cells in %parallel,
596  * making sure that only cells are worked on that do not share any degree of
597  * freedom (this makes the loop thread-safe when writing into destination
598  * vectors). This is a more advanced strategy compared to the WorkStream
599  * class described in the @ref threads module. Of course, to not destroy
600  * thread-safety, we have to be careful when writing into class-global
601  * structures.
602  *
603 
604  *
605  * The class implementing the Laplace operator has three template arguments,
606  * one for the dimension (as many deal.II classes carry), one for the degree
607  * of the finite element (which we need to enable efficient computations
608  * through the FEEvaluation class), and one for the underlying scalar
609  * type. We want to use <code>double</code> numbers (i.e., double precision,
610  * 64-bit floating point) for the final matrix, but floats (single
611  * precision, 32-bit floating point numbers) for the multigrid level
612  * matrices (as that is only a preconditioner, and floats can be processed
613  * twice as fast). The class FEEvaluation also takes a template argument for
614  * the number of quadrature points in one dimension. In the code below, we
615  * hard-code it to <code>fe_degree+1</code>. If we wanted to change it
616  * independently of the polynomial degree, we would need to add a template
617  * parameter as is done in the MatrixFreeOperators::LaplaceOperator class.
618  *
619 
620  *
621  * As a sidenote, if we implemented several different operations on the same
622  * grid and degrees of freedom (like a mass matrix and a Laplace matrix), we
623  * would define two classes like the current one for each of the operators
624  * (derived from the MatrixFreeOperators::Base class), and let both of them
625  * refer to the same MatrixFree data cache from the general problem
626  * class. The interface through MatrixFreeOperators::Base requires us to
627  * only provide a minimal set of functions. This concept allows for writing
628  * complex application codes with many matrix-free operations.
629  *
630 
631  *
632  * @note Storing values of type <code>VectorizedArray<number></code>
633  * requires care: Here, we use the deal.II table class which is prepared to
634  * hold the data with correct alignment. However, storing e.g. an
635  * <code>std::vector<VectorizedArray<number> ></code> is not possible with
636  * vectorization: A certain alignment of the data with the memory address
637  * boundaries is required (essentially, a VectorizedArray that is 32 bytes
638  * long in case of AVX needs to start at a memory address that is divisible
639  * by 32). The table class (as well as the AlignedVector class it is based
640  * on) makes sure that this alignment is respected, whereas std::vector does
641  * not in general, which may lead to segmentation faults at strange places
642  * for some systems or suboptimal performance for other systems.

747  * <a name="LocalevaluationofLaplaceoperator"></a>
748  * <h4>Local evaluation of Laplace operator</h4>
749  *
750 
751  *
752  * Here comes the main function of this class, the evaluation of the
753  * matrix-vector product (or, in general, a finite element operator
754  * evaluation). This is done in a function that takes exactly four
755  * arguments, the MatrixFree object, the destination and source vectors, and
756  * a range of cells that are to be worked on. The method
757  * <code>cell_loop</code> in the MatrixFree class will internally call this
758  * function with some range of cells that is obtained by checking which
759  * cells are possible to work on simultaneously so that write operations do
760  * not cause any race condition. Note that the cell range used in the loop
761  * is not directly the number of (active) cells in the current mesh, but
762  * rather a collection of batches of cells. In other word, "cell" may be
763  * the wrong term to begin with, since FEEvaluation groups data from several
764  * cells together. This means that in the loop over quadrature points we are
765  * actually seeing a group of quadrature points of several cells as one
766  * block. This is done to enable a higher degree of vectorization. The
767  * number of such "cells" or "cell batches" is stored in MatrixFree and can
768  * be queried through MatrixFree::n_cell_batches(). Compared to the deal.II
769  * cell iterators, in this class all cells are laid out in a plain array
770  * with no direct knowledge of level or neighborship relations, which makes
771  * it possible to index the cells by unsigned integers.
772  *
773 
774  *
775  * The implementation of the Laplace operator is quite simple: First, we
776  * need to create an object FEEvaluation that contains the computational
777  * kernels and has data fields to store temporary results (e.g. gradients
778  * evaluated on all quadrature points on a collection of a few cells). Note
779  * that temporary results do not use a lot of memory, and since we specify
780  * template arguments with the element order, the data is stored on the
781  * stack (without expensive memory allocation). Usually, one only needs to
782  * set two template arguments, the dimension as a first argument and the
783  * degree of the finite element as the second argument (this is equal to the
784  * number of degrees of freedom per dimension minus one for FE_Q
785  * elements). However, here we also want to be able to use float numbers for
786  * the multigrid preconditioner, which is the last (fifth) template
787  * argument. Therefore, we cannot rely on the default template arguments and
788  * must also fill the third and fourth field, consequently. The third
789  * argument specifies the number of quadrature points per direction and has
790  * a default value equal to the degree of the element plus one. The fourth
791  * argument sets the number of components (one can also evaluate
792  * vector-valued functions in systems of PDEs, but the default is a scalar
793  * element), and finally the last argument sets the number type.
794  *
795 
796  *
797  * Next, we loop over the given cell range and then we continue with the
798  * actual implementation: <ol> <li>Tell the FEEvaluation object the (macro)
799  * cell we want to work on. <li>Read in the values of the source vectors
800  * (@p read_dof_values), including the resolution of constraints. This
801  * stores @f$u_\mathrm{cell}@f$ as described in the introduction. <li>Compute
802  * the unit-cell gradient (the evaluation of finite element
803  * functions). Since FEEvaluation can combine value computations with
804  * gradient computations, it uses a unified interface to all kinds of
805  * derivatives of order between zero and two. We only want gradients, no
806  * values and no second derivatives, so we set the function arguments to
807  * true in the gradient slot (second slot), and to false in the values slot
808  * (first slot). There is also a third slot for the Hessian which is
809  * false by default, so it needs not be given. Note that the FEEvaluation
810  * class internally evaluates shape functions in an efficient way where one
811  * dimension is worked on at a time (using the tensor product form of shape
812  * functions and quadrature points as mentioned in the introduction). This
813  * gives complexity equal to @f$\mathcal O(d^2 (p+1)^{d+1})@f$ for polynomial
814  * degree @f$p@f$ in @f$d@f$ dimensions, compared to the naive approach with loops
815  * over all local degrees of freedom and quadrature points that is used in
816  * FEValues and costs @f$\mathcal O(d (p+1)^{2d})@f$. <li>Next comes the
817  * application of the Jacobian transformation, the multiplication by the
818  * variable coefficient and the quadrature weight. FEEvaluation has an
819  * access function @p get_gradient that applies the Jacobian and returns the
820  * gradient in real space. Then, we just need to multiply by the (scalar)
821  * coefficient, and let the function @p submit_gradient apply the second
822  * Jacobian (for the test function) and the quadrature weight and Jacobian
823  * determinant (JxW). Note that the submitted gradient is stored in the same
824  * data field as where it is read from in @p get_gradient. Therefore, you
825  * need to make sure to not read from the same quadrature point again after
826  * having called @p submit_gradient on that particular quadrature point. In
827  * general, it is a good idea to copy the result of @p get_gradient when it
828  * is used more often than once. <li>Next follows the summation over
829  * quadrature points for all test functions that corresponds to the actual
830  * integration step. For the Laplace operator, we just multiply by the
831  * gradient, so we call the integrate function with the respective argument
832  * set. If you have an equation where you test by both the values of the
833  * test functions and the gradients, both template arguments need to be set
834  * to true. Calling first the integrate function for values and then
835  * gradients in a separate call leads to wrong results, since the second
836  * call will internally overwrite the results from the first call. Note that
837  * there is no function argument for the second derivative for integrate
838  * step. <li>Eventually, the local contributions in the vector
839  * @f$v_\mathrm{cell}@f$ as mentioned in the introduction need to be added into
840  * the result vector (and constraints are applied). This is done with a call
841  * to @p distribute_local_to_global, the same name as the corresponding
842  * function in the AffineConstraints (only that we now store the local vector
843  * in the FEEvaluation object, as are the indices between local and global
844  * degrees of freedom). </ol>

846  * @code
847  * template <int dim, int fe_degree, typename number>

864  * for (unsigned int q = 0; q < phi.n_q_points; ++q)

874  *
875  * This function implements the loop over all cells for the
876  * Base::apply_add() interface. This is done with the @p cell_loop of the
877  * MatrixFree class, which takes the operator() of this class with arguments
878  * MatrixFree, OutVector, InVector, cell_range. When working with MPI
879  * parallelization (but no threading) as is done in this tutorial program,
880  * the cell loop corresponds to the following three lines of code:
881  *
882 
883  *
884  * <div class=CodeFragmentInTutorialComment>
885  * @code
886  * src.update_ghost_values();
887  * local_apply(*this->data, dst, src, std::make_pair(0U,
888  * data.n_cell_batches()));
889  * dst.compress(VectorOperation::add);
890  * @endcode
891  * </div>
892  *
893 
894  *
895  * Here, the two calls update_ghost_values() and compress() perform the data
896  * exchange on processor boundaries for MPI, once for the source vector
897  * where we need to read from entries owned by remote processors, and once
898  * for the destination vector where we have accumulated parts of the
899  * residuals that need to be added to the respective entry of the owner
900  * processor. However, MatrixFree::cell_loop does not only abstract away
901  * those two calls, but also performs some additional optimizations. On the
902  * one hand, it will split the update_ghost_values() and compress() calls in
903  * a way to allow for overlapping communication and computation. The
904  * local_apply function is then called with three cell ranges representing
905  * partitions of the cell range from 0 to MatrixFree::n_cell_batches(). On
906  * the other hand, cell_loop also supports thread parallelism in which case
907  * the cell ranges are split into smaller chunks and scheduled in an
908  * advanced way that avoids access to the same vector entry by several
909  * threads. That feature is explained in @ref step_48 "step-48".
910  *
911 
912  *
913  * Note that after the cell loop, the constrained degrees of freedom need to
914  * be touched once more for sensible vmult() operators: Since the assembly
915  * loop automatically resolves constraints (just as the
916  * AffineConstraints::distribute_local_to_global() call does), it does not
917  * compute any contribution for constrained degrees of freedom, leaving the
918  * respective entries zero. This would represent a matrix that had empty
919  * rows and columns for constrained degrees of freedom. However, iterative
920  * solvers like CG only work for non-singular matrices. The easiest way to
921  * do that is to set the sub-block of the matrix that corresponds to
922  * constrained DoFs to an identity matrix, in which case application of the
923  * matrix would simply copy the elements of the right hand side vector into
924  * the left hand side. Fortunately, the vmult() implementations
925  * MatrixFreeOperators::Base do this automatically for us outside the
926  * apply_add() function, so we do not need to take further action here.
927  *
928 
929  *
930  * When using the combination of MatrixFree and FEEvaluation in parallel
931  * with MPI, there is one aspect to be careful about &mdash; the indexing
932  * used for accessing the vector. For performance reasons, MatrixFree and
933  * FEEvaluation are designed to access vectors in MPI-local index space also
934  * when working with multiple processors. Working in local index space means
935  * that no index translation needs to be performed at the place the vector
936  * access happens, apart from the unavoidable indirect addressing. However,
937  * local index spaces are ambiguous: While it is standard convention to
938  * access the locally owned range of a vector with indices between 0 and the
939  * local size, the numbering is not so clear for the ghosted entries and
940  * somewhat arbitrary. For the matrix-vector product, only the indices
941  * appearing on locally owned cells (plus those referenced via hanging node
942  * constraints) are necessary. However, in deal.II we often set all the
943  * degrees of freedom on ghosted elements as ghosted vector entries, called
944  * the @ref GlossLocallyRelevantDof "locally relevant DoFs described in the
945  * glossary". In that case, the MPI-local index of a ghosted vector entry
946  * can in general be different in the two possible ghost sets, despite
947  * referring to the same global index. To avoid problems, FEEvaluation
948  * checks that the partitioning of the vector used for the matrix-vector
949  * product does indeed match with the partitioning of the indices in
950  * MatrixFree by a check called
951  * LinearAlgebra::distributed::Vector::partitioners_are_compatible. To
952  * facilitate things, the MatrixFreeOperators::Base class includes a
953  * mechanism to fit the ghost set to the correct layout. This happens in the
954  * ghost region of the vector, so keep in mind that the ghost region might
955  * be modified in both the destination and source vector after a call to a
956  * vmult() method. This is legitimate because the ghost region of a
957  * distributed deal.II vector is a mutable section and filled on
958  * demand. Vectors used in matrix-vector products must not be ghosted upon
959  * entry of vmult() functions, so no information gets lost.
960  *
961  * @code
962  * template <int dim, int fe_degree, typename number>
963  * void LaplaceOperator<dim, fe_degree, number>::apply_add(
964  * LinearAlgebra::distributed::Vector<number> & dst,

972  * @endcode
973  *
974  * The following function implements the computation of the diagonal of the
975  * operator. Computing matrix entries of a matrix-free operator evaluation
976  * turns out to be more complicated than evaluating the
977  * operator. Fundamentally, we could obtain a matrix representation of the
978  * operator by applying the operator on <i>all</i> unit vectors. Of course,
979  * that would be very inefficient since we would need to perform <i>n</i>
980  * operator evaluations to retrieve the whole matrix. Furthermore, this
981  * approach would completely ignore the matrix sparsity. On an individual
982  * cell, however, this is the way to go and actually not that inefficient as
983  * there usually is a coupling between all degrees of freedom inside the
984  * cell.
985  *
986 
987  *
988  * We first initialize the diagonal vector to the correct parallel
989  * layout. This vector is encapsulated in a member called
990  * inverse_diagonal_entries of type DiagonalMatrix in the base class
991  * MatrixFreeOperators::Base. This member is a shared pointer that we first
992  * need to initialize and then get the vector representing the diagonal
993  * entries in the matrix. As to the actual diagonal computation, we again
994  * use the cell_loop infrastructure of MatrixFree to invoke a local worker
995  * routine called local_compute_diagonal(). Since we will only write into a
996  * vector but not have any source vector, we put a dummy argument of type
997  * <tt>unsigned int</tt> in place of the source vector to confirm with the
998  * cell_loop interface. After the loop, we need to set the vector entries
999  * subject to Dirichlet boundary conditions to one (either those on the
1000  * boundary described by the AffineConstraints object inside MatrixFree or
1001  * the indices at the interface between different grid levels in adaptive
1002  * multigrid). This is done through the function
1003  * MatrixFreeOperators::Base::set_constrained_entries_to_one() and matches
1004  * with the setting in the matrix-vector product provided by the Base
1005  * operator. Finally, we need to invert the diagonal entries which is the
1006  * form required by the Chebyshev smoother based on the Jacobi iteration. In
1007  * the loop, we assert that all entries are non-zero, because they should
1008  * either have obtained a positive contribution from integrals or be
1009  * constrained and treated by @p set_constrained_entries_to_one() following
1010  * cell_loop.
1011  *
1012  * @code
1013  * template <int dim, int fe_degree, typename number>
1014  * void LaplaceOperator<dim, fe_degree, number>::compute_diagonal()
1015  * {
1016  * this->inverse_diagonal_entries.reset(
1017  * new DiagonalMatrix<LinearAlgebra::distributed::Vector<number>>());
1018  * LinearAlgebra::distributed::Vector<number> &inverse_diagonal =
1019  * this->inverse_diagonal_entries->get_vector();
1020  * this->data->initialize_dof_vector(inverse_diagonal);
1021  * unsigned int dummy = 0;
1022  * this->data->cell_loop(&LaplaceOperator::local_compute_diagonal,
1023  * this,
1024  * inverse_diagonal,
1025  * dummy);
1026  *
1027  * this->set_constrained_entries_to_one(inverse_diagonal);
1028  *
1029  * for (unsigned int i = 0; i < inverse_diagonal.locally_owned_size(); ++i)
1030  * {
1031  * Assert(inverse_diagonal.local_element(i) > 0.,
1032  * ExcMessage("No diagonal entry in a positive definite operator "
1033  * "should be zero"));
1034  * inverse_diagonal.local_element(i) =
1035  * 1. / inverse_diagonal.local_element(i);
1036  * }
1037  * }

1039  *
1040  *
1041  * @endcode
1042  *
1043  * In the local compute loop, we compute the diagonal by a loop over all
1044  * columns in the local matrix and putting the entry 1 in the <i>i</i>th
1045  * slot and a zero entry in all other slots, i.e., we apply the cell-wise
1046  * differential operator on one unit vector at a time. The inner part
1047  * invoking FEEvaluation::evaluate, the loop over quadrature points, and
1048  * FEEvalution::integrate, is exactly the same as in the local_apply
1049  * function. Afterwards, we pick out the <i>i</i>th entry of the local
1050  * result and put it to a temporary storage (as we overwrite all entries in
1051  * the array behind FEEvaluation::get_dof_value() with the next loop
1052  * iteration). Finally, the temporary storage is written to the destination
1053  * vector. Note how we use FEEvaluation::get_dof_value() and
1054  * FEEvaluation::submit_dof_value() to read and write to the data field that
1055  * FEEvaluation uses for the integration on the one hand and writes into the
1056  * global vector on the other hand.
1057  *
1058 
1059  *
1060  * Given that we are only interested in the matrix diagonal, we simply throw
1061  * away all other entries of the local matrix that have been computed along
1062  * the way. While it might seem wasteful to compute the complete cell matrix
1063  * and then throw away everything but the diagonal, the integration are so
1064  * efficient that the computation does not take too much time. Note that the
1065  * complexity of operator evaluation per element is @f$\mathcal
1066  * O((p+1)^{d+1})@f$ for polynomial degree @f$k@f$, so computing the whole matrix
1067  * costs us @f$\mathcal O((p+1)^{2d+1})@f$ operations, not too far away from
1068  * @f$\mathcal O((p+1)^{2d})@f$ complexity for computing the diagonal with
1069  * FEValues. Since FEEvaluation is also considerably faster due to
1070  * vectorization and other optimizations, the diagonal computation with this
1071  * function is actually the fastest (simple) variant. (It would be possible
1072  * to compute the diagonal with sum factorization techniques in @f$\mathcal
1073  * O((p+1)^{d+1})@f$ operations involving specifically adapted
1074  * kernels&mdash;but since such kernels are only useful in that particular
1075  * context and the diagonal computation is typically not on the critical
1076  * path, they have not been implemented in deal.II.)
1077  *
1078 
1079  *
1080  * Note that the code that calls distribute_local_to_global on the vector to
1081  * accumulate the diagonal entries into the global matrix has some
1082  * limitations. For operators with hanging node constraints that distribute
1083  * an integral contribution of a constrained DoF to several other entries
1084  * inside the distribute_local_to_global call, the vector interface used
1085  * here does not exactly compute the diagonal entries, but lumps some
1086  * contributions located on the diagonal of the local matrix that would end
1087  * up in a off-diagonal position of the global matrix to the diagonal. The
1088  * result is correct up to discretization accuracy as explained in <a
1089  * href="http://dx.doi.org/10.4208/cicp.101214.021015a">Kormann (2016),
1090  * section 5.3</a>, but not mathematically equal. In this tutorial program,
1091  * no harm can happen because the diagonal is only used for the multigrid
1092  * level matrices where no hanging node constraints appear.
1093  *
1094  * @code

1458  * <a name="LaplaceProblemsolve"></a>
1459  * <h4>LaplaceProblem::solve</h4>
1460  *
1461 
1462  *
1463  * The solution process is similar as in @ref step_16 "step-16". We start with the setup of
1464  * the transfer. For LinearAlgebra::distributed::Vector, there is a very
1465  * fast transfer class called MGTransferMatrixFree that does the
1466  * interpolation between the grid levels with the same fast sum
1467  * factorization kernels that get also used in FEEvaluation.
1468  *
1469  * @code
1470  * template <int dim>
1471  * void LaplaceProblem<dim>::solve()
1472  * {
1473  * Timer time;
1474  * MGTransferMatrixFree<dim, float> mg_transfer(mg_constrained_dofs);
1475  * mg_transfer.build(dof_handler);
1476  * setup_time += time.wall_time();
1477  * time_details << "MG build transfer time (CPU/wall) " << time.cpu_time()
1478  * << "s/" << time.wall_time() << "s\n";
1479  * time.restart();
1480  *
1481  * @endcode
1482  *
1483  * As a smoother, this tutorial program uses a Chebyshev iteration instead
1484  * of SOR in @ref step_16 "step-16". (SOR would be very difficult to implement because we
1485  * do not have the matrix elements available explicitly, and it is
1486  * difficult to make it work efficiently in %parallel.) The smoother is
1487  * initialized with our level matrices and the mandatory additional data
1488  * for the Chebyshev smoother. We use a relatively high degree here (5),
1489  * since matrix-vector products are comparably cheap. We choose to smooth
1490  * out a range of @f$[1.2 \hat{\lambda}_{\max}/15,1.2 \hat{\lambda}_{\max}]@f$
1491  * in the smoother where @f$\hat{\lambda}_{\max}@f$ is an estimate of the
1492  * largest eigenvalue (the factor 1.2 is applied inside
1493  * PreconditionChebyshev). In order to compute that eigenvalue, the
1494  * Chebyshev initialization performs a few steps of a CG algorithm
1495  * without preconditioner. Since the highest eigenvalue is usually the
1496  * easiest one to find and a rough estimate is enough, we choose 10
1497  * iterations. Finally, we also set the inner preconditioner type in the
1498  * Chebyshev method which is a Jacobi iteration. This is represented by
1499  * the DiagonalMatrix class that gets the inverse diagonal entry provided
1500  * by our LaplaceOperator class.
1501  *
1502 
1503  *
1504  * On level zero, we initialize the smoother differently because we want
1505  * to use the Chebyshev iteration as a solver. PreconditionChebyshev
1506  * allows the user to switch to solver mode where the number of iterations
1507  * is internally chosen to the correct value. In the additional data
1508  * object, this setting is activated by choosing the polynomial degree to
1509  * @p numbers::invalid_unsigned_int. The algorithm will then attack all
1510  * eigenvalues between the smallest and largest one in the coarse level
1511  * matrix. The number of steps in the Chebyshev smoother are chosen such
1512  * that the Chebyshev convergence estimates guarantee to reduce the
1513  * residual by the number specified in the variable @p
1514  * smoothing_range. Note that for solving, @p smoothing_range is a
1515  * relative tolerance and chosen smaller than one, in this case, we select
1516  * three orders of magnitude, whereas it is a number larger than 1 when
1517  * only selected eigenvalues are smoothed.
1518  *
1519 
1520  *
1521  * From a computational point of view, the Chebyshev iteration is a very
1522  * attractive coarse grid solver as long as the coarse size is
1523  * moderate. This is because the Chebyshev method performs only
1524  * matrix-vector products and vector updates, which typically parallelize
1525  * better to the largest cluster size with more than a few tens of
1526  * thousands of cores than inner product involved in other iterative
1527  * methods. The former involves only local communication between neighbors
1528  * in the (coarse) mesh, whereas the latter requires global communication
1529  * over all processors.
1530  *
1531  * @code
1532  * using SmootherType =
1533  * PreconditionChebyshev<LevelMatrixType,
1534  * LinearAlgebra::distributed::Vector<float>>;
1535  * mg::SmootherRelaxation<SmootherType,
1536  * LinearAlgebra::distributed::Vector<float>>
1537  * mg_smoother;
1538  * MGLevelObject<typename SmootherType::AdditionalData> smoother_data;
1539  * smoother_data.resize(0, triangulation.n_global_levels() - 1);
1540  * for (unsigned int level = 0; level < triangulation.n_global_levels();
1541  * ++level)
1542  * {
1543  * if (level > 0)
1544  * {
1545  * smoother_data[level].smoothing_range = 15.;
1546  * smoother_data[level].degree = 5;
1547  * smoother_data[level].eig_cg_n_iterations = 10;
1548  * }
1549  * else
1550  * {
1551  * smoother_data[0].smoothing_range = 1e-3;
1552  * smoother_data[0].degree = numbers::invalid_unsigned_int;
1553  * smoother_data[0].eig_cg_n_iterations = mg_matrices[0].m();
1554  * }
1555  * mg_matrices[level].compute_diagonal();
1556  * smoother_data[level].preconditioner =
1557  * mg_matrices[level].get_matrix_diagonal_inverse();
1558  * }
1559  * mg_smoother.initialize(mg_matrices, smoother_data);
1560  *
1561  * MGCoarseGridApplySmoother<LinearAlgebra::distributed::Vector<float>>
1562  * mg_coarse;
1563  * mg_coarse.initialize(mg_smoother);
1564  *
1565  * @endcode
1566  *
1567  * The next step is to set up the interface matrices that are needed for the
1568  * case with hanging nodes. The adaptive multigrid realization in deal.II
1569  * implements an approach called local smoothing. This means that the
1570  * smoothing on the finest level only covers the local part of the mesh
1571  * defined by the fixed (finest) grid level and ignores parts of the
1572  * computational domain where the terminal cells are coarser than this
1573  * level. As the method progresses to coarser levels, more and more of the
1574  * global mesh will be covered. At some coarser level, the whole mesh will
1575  * be covered. Since all level matrices in the multigrid method cover a
1576  * single level in the mesh, no hanging nodes appear on the level matrices.
1577  * At the interface between multigrid levels, homogeneous Dirichlet boundary
1578  * conditions are set while smoothing. When the residual is transferred to
1579  * the next coarser level, however, the coupling over the multigrid
1580  * interface needs to be taken into account. This is done by the so-called
1581  * interface (or edge) matrices that compute the part of the residual that
1582  * is missed by the level matrix with
1583  * homogeneous Dirichlet conditions. We refer to the @ref mg_paper
1584  * "Multigrid paper by Janssen and Kanschat" for more details.
1585  *
1586 
1587  *
1588  * For the implementation of those interface matrices, there is already a
1589  * pre-defined class MatrixFreeOperators::MGInterfaceOperator that wraps
1590  * the routines MatrixFreeOperators::Base::vmult_interface_down() and
1591  * MatrixFreeOperators::Base::vmult_interface_up() in a new class with @p
1592  * vmult() and @p Tvmult() operations (that were originally written for
1593  * matrices, hence expecting those names). Note that vmult_interface_down
1594  * is used during the restriction phase of the multigrid V-cycle, whereas
1595  * vmult_interface_up is used during the prolongation phase.
1596  *
1597 
1598  *
1599  * Once the interface matrix is created, we set up the remaining Multigrid
1600  * preconditioner infrastructure in complete analogy to @ref step_16 "step-16" to obtain
1601  * a @p preconditioner object that can be applied to a matrix.
1602  *
1603  * @code
1604  * mg::Matrix<LinearAlgebra::distributed::Vector<float>> mg_matrix(
1605  * mg_matrices);
1606  *
1607  * MGLevelObject<MatrixFreeOperators::MGInterfaceOperator<LevelMatrixType>>
1608  * mg_interface_matrices;
1609  * mg_interface_matrices.resize(0, triangulation.n_global_levels() - 1);
1610  * for (unsigned int level = 0; level < triangulation.n_global_levels();
1611  * ++level)
1612  * mg_interface_matrices[level].initialize(mg_matrices[level]);
1613  * mg::Matrix<LinearAlgebra::distributed::Vector<float>> mg_interface(
1614  * mg_interface_matrices);
1615  *
1616  * Multigrid<LinearAlgebra::distributed::Vector<float>> mg(
1617  * mg_matrix, mg_coarse, mg_transfer, mg_smoother, mg_smoother);
1618  * mg.set_edge_matrices(mg_interface, mg_interface);
1619  *
1620  * PreconditionMG<dim,
1621  * LinearAlgebra::distributed::Vector<float>,
1622  * MGTransferMatrixFree<dim, float>>
1623  * preconditioner(dof_handler, mg, mg_transfer);
1624  *
1625  * @endcode
1626  *
1627  * The setup of the multigrid routines is quite easy and one cannot see
1628  * any difference in the solve process as compared to @ref step_16 "step-16". All the
1629  * magic is hidden behind the implementation of the LaplaceOperator::vmult
1630  * operation. Note that we print out the solve time and the accumulated
1631  * setup time through standard out, i.e., in any case, whereas detailed
1632  * times for the setup operations are only printed in case the flag for
1633  * detail_times in the constructor is changed.
1634  *
1635 
1636  *
1637  *
1638  * @code
1639  * SolverControl solver_control(100, 1e-12 * system_rhs.l2_norm());

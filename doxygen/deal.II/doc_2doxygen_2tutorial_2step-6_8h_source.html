526  * <h4>Step6::assemble_system</h4>
527  *
528 
529  *
530  * Next, we have to assemble the matrix. However, to copy the local matrix and
531  * vector on each cell into the global system, we are no longer using a
532  * hand-written loop. Instead, we use
533  * AffineConstraints::distribute_local_to_global() that internally executes
534  * this loop while performing Gaussian elimination on rows and columns
535  * corresponding to constrained degrees on freedom.
536  *
537 
538  *
539  * The rest of the code that forms the local contributions remains
540  * unchanged. It is worth noting, however, that under the hood several things
541  * are different than before. First, the variable <code>dofs_per_cell</code>
542  * and return value of <code>quadrature_formula.size()</code> now are 9 each,
543  * where they were 4 before. Introducing such variables as abbreviations is a
544  * good strategy to make code work with different elements without having to
545  * change too much code. Secondly, the <code>fe_values</code> object of course
546  * needs to do other things as well, since the shape functions are now
547  * quadratic, rather than linear, in each coordinate variable. Again, however,
548  * this is something that is completely handled by the library.
549  *
550  * @code
551  * template <int dim>
552  * void Step6<dim>::assemble_system()
553  * {
554  * const QGauss<dim> quadrature_formula(fe.degree + 1);
555  *
556  * FEValues<dim> fe_values(fe,
557  * quadrature_formula,
558  * update_values | update_gradients |
559  * update_quadrature_points | update_JxW_values);
560  *
561  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
562  *
563  * FullMatrix<double> cell_matrix(dofs_per_cell, dofs_per_cell);
564  * Vector<double> cell_rhs(dofs_per_cell);
565  *
566  * std::vector<types::global_dof_index> local_dof_indices(dofs_per_cell);
567  *
568  * for (const auto &cell : dof_handler.active_cell_iterators())
569  * {
570  * cell_matrix = 0;
571  * cell_rhs = 0;
572  *
573  * fe_values.reinit(cell);
574  *

576  * {
577  * const double current_coefficient =
578  * coefficient(fe_values.quadrature_point(q_index));

580  * {
581  * for (const unsigned int j : fe_values.dof_indices())

589  * fe_values.shape_value(i, q_index) * // phi_i(x_q)
590  * fe_values.JxW(q_index)); // dx
591  * }
592  * }

623  * <a name="Step6solve"></a>
624  * <h4>Step6::solve</h4>
625  *
626 
627  *
628  * We continue with gradual improvements. The function that solves the linear
629  * system again uses the SSOR preconditioner, and is again unchanged except
630  * that we have to incorporate hanging node constraints. As mentioned above,
631  * the degrees of freedom from the AffineConstraints object corresponding to
632  * hanging node constraints and boundary values have been removed from the
633  * linear system by giving the rows and columns of the matrix a special
634  * treatment. This way, the values for these degrees of freedom have wrong,
635  * but well-defined values after solving the linear system. What we then have
636  * to do is to use the constraints to assign to them the values that they
637  * should have. This process, called <code>distributing</code> constraints,
638  * computes the values of constrained nodes from the values of the
639  * unconstrained ones, and requires only a single additional function call
640  * that you find at the end of this function:
641  *
642 
643  *
644  *
645  * @code
646  * template <int dim>
647  * void Step6<dim>::solve()
648  * {
649  * SolverControl solver_control(1000, 1e-12);
650  * SolverCG<Vector<double>> solver(solver_control);
651  *
652  * PreconditionSSOR<SparseMatrix<double>> preconditioner;
653  * preconditioner.initialize(system_matrix, 1.2);
654  *
655  * solver.solve(system_matrix, solution, system_rhs, preconditioner);
656  *
657  * constraints.distribute(solution);
658  * }
659  *
660  *
661  * @endcode

664  * <a name="Step6refine_grid"></a>
665  * <h4>Step6::refine_grid</h4>
666  *
667 
668  *
669  * We use a sophisticated error estimation scheme to refine the mesh instead
670  * of global refinement. We will use the KellyErrorEstimator class which
671  * implements an error estimator for the Laplace equation; it can in principle
672  * handle variable coefficients, but we will not use these advanced features,
673  * but rather use its most simple form since we are not interested in
674  * quantitative results but only in a quick way to generate locally refined
675  * grids.
676  *
677 
678  *
679  * Although the error estimator derived by Kelly et al. was originally
680  * developed for the Laplace equation, we have found that it is also well
681  * suited to quickly generate locally refined grids for a wide class of
682  * problems. This error estimator uses the solution gradient's jump at
683  * cell faces (which is a measure for the second derivatives) and
684  * scales it by the size of the cell. It is therefore a measure for the local
685  * smoothness of the solution at the place of each cell and it is thus
686  * understandable that it yields reasonable grids also for hyperbolic
687  * transport problems or the wave equation as well, although these grids are
688  * certainly suboptimal compared to approaches specially tailored to the
689  * problem. This error estimator may therefore be understood as a quick way to
690  * test an adaptive program.
691  *
692 
693  *
694  * The way the estimator works is to take a <code>DoFHandler</code> object
695  * describing the degrees of freedom and a vector of values for each degree of
696  * freedom as input and compute a single indicator value for each active cell
697  * of the triangulation (i.e. one value for each of the active cells). To do
698  * so, it needs two additional pieces of information: a face quadrature formula,
699  * i.e., a quadrature formula on <code>dim-1</code> dimensional objects. We use
700  * a 3-point Gauss rule again, a choice that is consistent and appropriate with
701  * the bi-quadratic finite element shape functions in this program.
702  * (What constitutes a suitable quadrature rule here of course depends on
703  * knowledge of the way the error estimator evaluates the solution field. As
704  * said above, the jump of the gradient is integrated over each face, which
705  * would be a quadratic function on each face for the quadratic elements in
706  * use in this example. In fact, however, it is the square of the jump of the
707  * gradient, as explained in the documentation of that class, and that is a
708  * quartic function, for which a 3 point Gauss formula is sufficient since it
709  * integrates polynomials up to order 5 exactly.)
710  *
711 
712  *
713  * Secondly, the function wants a list of boundary indicators for those
714  * boundaries where we have imposed Neumann values of the kind
715  * @f$\partial_n u(\mathbf x) = h(\mathbf x)@f$, along with a function @f$h(\mathbf
716  * x)@f$ for each such boundary. This information is represented by a map from
717  * boundary indicators to function objects describing the Neumann boundary
718  * values. In the present example program, we do not use Neumann boundary
719  * values, so this map is empty, and in fact constructed using the default
720  * constructor of the map in the place where the function call expects the
721  * respective function argument.
722  *
723 
724  *
725  * The output is a vector of values for all active cells. While it may
726  * make sense to compute the <b>value</b> of a solution degree of freedom
727  * very accurately, it is usually not necessary to compute the <b>error
728  * indicator</b> corresponding to the solution on a cell particularly
729  * accurately. We therefore typically use a vector of floats instead of a vector
730  * of doubles to represent error indicators.
731  *
732  * @code
733  * template <int dim>
734  * void Step6<dim>::refine_grid()
735  * {
736  * Vector<float> estimated_error_per_cell(triangulation.n_active_cells());
737  *
738  * KellyErrorEstimator<dim>::estimate(dof_handler,
739  * QGauss<dim - 1>(fe.degree + 1),
740  * {},
741  * solution,
742  * estimated_error_per_cell);
743  *
744  * @endcode
745  *
746  * The above function returned one error indicator value for each cell in
747  * the <code>estimated_error_per_cell</code> array. Refinement is now done
748  * as follows: refine those 30 per cent of the cells with the highest error
749  * values, and coarsen the 3 per cent of cells with the lowest values.
750  *
751 
752  *
753  * One can easily verify that if the second number were zero, this would
754  * approximately result in a doubling of cells in each step in two space
755  * dimensions, since for each of the 30 per cent of cells, four new would be
756  * replaced, while the remaining 70 per cent of cells remain untouched. In
757  * practice, some more cells are usually produced since it is disallowed
758  * that a cell is refined twice while the neighbor cell is not refined; in
759  * that case, the neighbor cell would be refined as well.
760  *
761 
762  *
763  * In many applications, the number of cells to be coarsened would be set to
764  * something larger than only three per cent. A non-zero value is useful
765  * especially if for some reason the initial (coarse) grid is already rather
766  * refined. In that case, it might be necessary to refine it in some
767  * regions, while coarsening in some other regions is useful. In our case
768  * here, the initial grid is very coarse, so coarsening is only necessary in
769  * a few regions where over-refinement may have taken place. Thus a small,
770  * non-zero value is appropriate here.
771  *
772 
773  *
774  * The following function now takes these refinement indicators and flags
775  * some cells of the triangulation for refinement or coarsening using the
776  * method described above. It is from a class that implements several
777  * different algorithms to refine a triangulation based on cell-wise error
778  * indicators.
779  *
780  * @code
781  * GridRefinement::refine_and_coarsen_fixed_number(triangulation,
782  * estimated_error_per_cell,
783  * 0.3,
784  * 0.03);
785  *
786  * @endcode
787  *
788  * After the previous function has exited, some cells are flagged for
789  * refinement, and some other for coarsening. The refinement or coarsening
790  * itself is not performed by now, however, since there are cases where
791  * further modifications of these flags is useful. Here, we don't want to do
792  * any such thing, so we can tell the triangulation to perform the actions
793  * for which the cells are flagged:
794  *
795  * @code
796  * triangulation.execute_coarsening_and_refinement();
797  * }
798  *
799  *
800  * @endcode

852  * <a name="Step6run"></a>
853  * <h4>Step6::run</h4>
854  *
855 
856  *
857  * The final function before <code>main()</code> is again the main driver of
858  * the class, <code>run()</code>. It is similar to the one of @ref step_5 "step-5", except
859  * that we generate a file in the program again instead of reading it from
860  * disk, in that we adaptively instead of globally refine the mesh, and that
861  * we output the solution on the final mesh in the present function.
862  *
863 
864  *
865  * The first block in the main loop of the function deals with mesh generation.
866  * If this is the first cycle of the program, instead of reading the grid from
867  * a file on disk as in the previous example, we now again create it using a
868  * library function. The domain is again a circle with center at the origin and
869  * a radius of one (these are the two hidden arguments to the function, which
870  * have default values).
871  *
872 
873  *
874  * You will notice by looking at the coarse grid that it is of inferior
875  * quality than the one which we read from the file in the previous example:
876  * the cells are less equally formed. However, using the library function this
877  * program works in any space dimension, which was not the case before.
878  *
879 
880  *
881  * In case we find that this is not the first cycle, we want to refine the
882  * grid. Unlike the global refinement employed in the last example program, we
883  * now use the adaptive procedure described above.
884  *
885 
886  *
887  * The rest of the loop looks as before:
888  *
889  * @code
890  * template <int dim>

897  * if (cycle == 0)

899  * GridGenerator::hyper_ball(triangulation);

901  * }
902  * else
903  * refine_grid();

908  *
909  * setup_system();
910  *
911  * std::cout << " Number of degrees of freedom: " << dof_handler.n_dofs()
912  * << std::endl;
913  *
914  * assemble_system();

924  * <a name="Thecodemaincodefunction"></a>
925  * <h3>The <code>main</code> function</h3>
926  *
927 
928  *
929  * The main function is unaltered in its functionality from the previous
930  * example, but we have taken a step of additional caution. Sometimes,
931  * something goes wrong (such as insufficient disk space upon writing an
932  * output file, not enough memory when trying to allocate a vector or a
933  * matrix, or if we can't read from or write to a file for whatever reason),
934  * and in these cases the library will throw exceptions. Since these are
935  * run-time problems, not programming errors that can be fixed once and for
936  * all, this kind of exceptions is not switched off in optimized mode, in
937  * contrast to the <code>Assert</code> macro which we have used to test
938  * against programming errors. If uncaught, these exceptions propagate the
939  * call tree up to the <code>main</code> function, and if they are not caught
940  * there either, the program is aborted. In many cases, like if there is not
941  * enough memory or disk space, we can't do anything but we can at least print
942  * some text trying to explain the reason why the program failed. A way to do
943  * so is shown in the following. It is certainly useful to write any larger
944  * program in this way, and you can do so by more or less copying this
945  * function except for the <code>try</code> block that actually encodes the
946  * functionality particular to the present application.
947  *
948  * @code
949  * int main()
950  * {
951  * @endcode
952  *
953  * The general idea behind the layout of this function is as follows: let's
954  * try to run the program as we did before...
955  *
956  * @code
957  * try
958  * {
959  * Step6<2> laplace_problem_2d;
960  * laplace_problem_2d.run();
961  * }
962  * @endcode
963  *
964  * ...and if this should fail, try to gather as much information as
965  * possible. Specifically, if the exception that was thrown is an object of
966  * a class that is derived from the C++ standard class
967  * <code>exception</code>, then we can use the <code>what</code> member
968  * function to get a string which describes the reason why the exception was
969  * thrown.
970  *
971 
972  *
973  * The deal.II exception classes are all derived from the standard class,
974  * and in particular, the <code>exc.what()</code> function will return
975  * approximately the same string as would be generated if the exception was
976  * thrown using the <code>Assert</code> macro. You have seen the output of
977  * such an exception in the previous example, and you then know that it
978  * contains the file and line number of where the exception occurred, and
979  * some other information. This is also what the following statements would
980  * print.
981  *
982 
983  *
984  * Apart from this, there isn't much that we can do except exiting the
985  * program with an error code (this is what the <code>return 1;</code>
986  * does):
987  *
988  * @code
989  * catch (std::exception &exc)
990  * {
991  * std::cerr << std::endl
992  * << std::endl
993  * << "----------------------------------------------------"
994  * << std::endl;
995  * std::cerr << "Exception on processing: " << std::endl
996  * << exc.what() << std::endl
997  * << "Aborting!" << std::endl
998  * << "----------------------------------------------------"
999  * << std::endl;
1000  *
1001  * return 1;
1002  * }
1003  * @endcode
1004  *
1005  * If the exception that was thrown somewhere was not an object of a class
1006  * derived from the standard <code>exception</code> class, then we can't do
1007  * anything at all. We then simply print an error message and exit.
1008  *
1009  * @code
1010  * catch (...)
1011  * {
1012  * std::cerr << std::endl
1013  * << std::endl
1014  * << "----------------------------------------------------"
1015  * << std::endl;
1016  * std::cerr << "Unknown exception!" << std::endl
1017  * << "Aborting!" << std::endl
1018  * << "----------------------------------------------------"
1019  * << std::endl;
1020  * return 1;
1021  * }
1022  *
1023  * @endcode
1024  *
1025  * If we got to this point, there was no exception which propagated up to
1026  * the main function (there may have been exceptions, but they were caught
1027  * somewhere in the program or the library). Therefore, the program
1028  * performed as was expected and we can return without error.
1029  *
1030  * @code
1031  * return 0;
1032  * }
1033  * @endcode

1313 为什么选择一个跟踪内部界面的网格是有意义的？有很多原因，但最重要的原因是我们在双线性表格中实际整合的内容。从概念上讲，我们想把 @f$A_{ij}^K=\int_K
1314 a(\mathbf x) \nabla \varphi_i(\mathbf x) \nabla \varphi_j(\mathbf x) ; dx@f$ 作为单元格 @f$K@f$ 对矩阵条目 @f$A_{ij}@f$ 的贡献来整合。我们无法精确计算，只能求助于正交法。我们知道，如果积分是平滑的，正交是准确的。这是因为正交法实质上是计算积分的多项式近似值，与积分在正交点上重合，然后计算这个多项式下的体积，作为原始积分下体积的近似值。如果积分在一个单元上是平滑的，这个多项式插值是准确的，但如果积分在一个单元上是不连续的，它通常是相当不准确的。
1315 
1316 因此，值得将细胞排列成这样的方式，使系数不连续的界面与细胞界面对齐。这样一来，系数在每个单元上都是恒定的，之后积分将是平滑的，其多项式近似和积分的正交近似都将是准确的。请注意，这样的排列方式在许多实际案例中很常见，因此deal.II提供了一些函数（如 @ref GlossMaterialId "material_id"）来帮助管理这种情况。请参考步骤28和步骤46，了解如何应用material ids的例子。

1320 最后说明一下：前面几段的讨论表明，我们在这里有一种非常具体的方式来说明我们认为的好的网格--它应该与系数中的跳跃相一致。但人们也可以在一个更普遍的环境中提出这样的问题。给定一些具有光滑解和光滑系数的方程，我们能说一个好的网格是什么样子的吗？这个问题的答案在直觉上比数学上更容易表述。一个好的网格，其单元格大体上都像正方形（或立方体，在三维空间）。一个不好的网格会包含一些在某些方向上非常细长的单元，或者，更广泛地说，其中有一些单元的边缘既短又长。有很多方法可以给每个单元分配一个数字质量指数，以衡量该单元是 "好 "还是 "坏"；其中一些经常被选择，因为它们便宜且容易计算，而另一些则是基于收敛性证明中的内容。前者的一个例子是一个单元格的最长边与最短边的比率。在理想的情况下，这个比率是1；不好的单元格的值远远大于1。后者的例子是考虑从参考单元 @f$\hat K=[0,1]^d@f$ 到实际单元 @f$K@f$ 的映射的梯度（"Jacobian"）；这个梯度是一个矩阵，进入误差估计的一个量是参考单元上所有点的最大值，即这个矩阵的最大和最小的特征值的比率。同样不难看出，如果单元格 @f$K@f$ 是 @f$\hat K@f$ 的仿生图像，这个比率是恒定的，对于正方形和长方体来说是一个。

1330 从数学的角度来看，拉普拉斯方程的解决方案
1331

1333

1335  -\Delta u = f
1336

1338 @f]
1339 
1340 在光滑有界的凸域上，已知其本身是光滑的。确切的光滑程度，即解所处的函数空间，取决于域的边界到底有多光滑，以及右手边的光滑程度。边界处可能会失去解的某些规律性，但一般来说，在域的紧凑子集中，解的可微性比右手边多一倍。特别是如果右手边满足 @f$f\in C^\infty(\Omega)@f$ ，那么 @f$u \in C^\infty(\Omega_i)@f$ ，其中 @f$\Omega_i@f$ 是 @f$\Omega@f$ 的任何紧凑子集（ @f$\Omega@f$ 是一个开放域，所以紧凑子集需要与 @f$\partial\Omega@f$ 保持一个正距离）。
1341 
1342 然而，我们为目前的例子选择的情况是不同的：我们看的是一个具有非常数系数的方程 @f$a(\mathbf x)@f$ 。
1343 
1344 @f[
1345 
1346 
1347  -\nabla \cdot (a \nabla u) = f.
1348 
1349 
1350 @f]
1351 
1352 这里，如果 @f$a@f$ 不光滑，那么无论 @f$f@f$ 如何，解也不会光滑。特别是，我们希望在 @f$a@f$ 沿直线（或沿三维平面）不连续的地方，解会有一个结点。这很容易看出来：例如，如果 @f$f@f$ 是连续的，那么 @f$f=-\nabla \cdot (a \nabla u)@f$ 也需要是连续的。这意味着 @f$a \nabla u@f$ 必须是可连续微分的（不存在扭结）。因此，如果 @f$a@f$ 有一个不连续，那么 @f$\nabla u@f$ 必须有一个相反的不连续，从而使两者完全抵消，它们的乘积得到一个没有不连续的函数。但是要使 @f$\nabla u@f$ 有一个不连续， @f$u@f$ 必须有一个结点。当然，这正是当前例子中所发生的情况，在解的图片中也很容易观察到。
1353 
1354 一般来说，如果系数 @f$a(\mathbf x)@f$ 沿着2D的直线或3D的平面是不连续的，那么解可能有一个结点，但解的梯度不会到无限大。这意味着，解至少还在<a href="https://en.wikipedia.org/wiki/Sobolev_space">Sobolev space</a> @f$W^{1,\infty}@f$ 中（也就是说，大致上是在导数有界的函数空间中）。另一方面，我们知道，在最极端的情况下--即域有重入角，右手边只满足 @f$f\in H^{-1}@f$ ，或者系数 @f$a@f$ 只在 @f$L^\infty@f$ 中--我们所能期望的是， @f$u\in H^1@f$ （即导数是可平方整除的函数的<a
1355 href="https://en.wikipedia.org/wiki/Sobolev_space#Sobolev_spaces_with_integer_k">Sobolev
1356 space</a>），是比 @f$W^{1,\infty}@f$ 大很多的空间 。要创造出解在空间 @f$H^{1+s}@f$ 中的案例并不十分困难，我们可以让 @f$s@f$ 变得像我们想要的那样小。这样的情况经常被用来测试自适应有限元方法，因为网格要解决导致解不再在 @f$W^{1,\infty}@f$ 中的奇异点。
1357 
1358 人们为此使用的典型例子叫做<i>Kellogg problem</i>（指 @cite Kel74 ），在常用的形式中，它的系数 @f$a(\mathbf x)@f$ 在平面的四个象限有不同的值（或在 @f${\mathbb R}^3@f$ 的八个象限有不同的值）。确切的规则性程度（上述索博列夫空间索引中的 @f$s@f$ ）取决于 @f$a(\mathbf x)@f$ 的值在原点处聚集，通过选择足够大的跳跃，可以使解的规则性尽可能地接近 @f$H^1@f$ 。

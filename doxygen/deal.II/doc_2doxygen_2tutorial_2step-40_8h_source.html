761  * <a name="LaplaceProblemoutput_results"></a>
762  * <h4>LaplaceProblem::output_results</h4>
763  *
764 
765  *
766  * Compared to the corresponding function in @ref step_6 "step-6", the one here is a tad
767  * more complicated. There are two reasons: the first one is that we do not
768  * just want to output the solution but also for each cell which processor
769  * owns it (i.e. which "subdomain" it is in). Secondly, as discussed at
770  * length in @ref step_17 "step-17" and @ref step_18 "step-18", generating graphical data can be a
771  * bottleneck in parallelizing. In @ref step_18 "step-18", we have moved this step out of
772  * the actual computation but shifted it into a separate program that later
773  * combined the output from various processors into a single file. But this
774  * doesn't scale: if the number of processors is large, this may mean that
775  * the step of combining data on a single processor later becomes the
776  * longest running part of the program, or it may produce a file that's so
777  * large that it can't be visualized any more. We here follow a more
778  * sensible approach, namely creating individual files for each MPI process
779  * and leaving it to the visualization program to make sense of that.
780  *
781 
782  *
783  * To start, the top of the function looks like it usually does. In addition
784  * to attaching the solution vector (the one that has entries for all locally
785  * relevant, not only the locally owned, elements), we attach a data vector
786  * that stores, for each cell, the subdomain the cell belongs to. This is
787  * slightly tricky, because of course not every processor knows about every
788  * cell. The vector we attach therefore has an entry for every cell that the
789  * current processor has in its mesh (locally owned ones, ghost cells, and
790  * artificial cells), but the DataOut class will ignore all entries that
791  * correspond to cells that are not owned by the current processor. As a
792  * consequence, it doesn't actually matter what values we write into these
793  * vector entries: we simply fill the entire vector with the number of the
794  * current MPI process (i.e. the subdomain_id of the current process); this
795  * correctly sets the values we care for, i.e. the entries that correspond
796  * to locally owned cells, while providing the wrong value for all other
797  * elements -- but these are then ignored anyway.
798  *
799  * @code
800  * template <int dim>
801  * void LaplaceProblem<dim>::output_results(const unsigned int cycle) const
802  * {
803  * DataOut<dim> data_out;
804  * data_out.attach_dof_handler(dof_handler);
805  * data_out.add_data_vector(locally_relevant_solution, "u");
806  *
807  * Vector<float> subdomain(triangulation.n_active_cells());
808  * for (unsigned int i = 0; i < subdomain.size(); ++i)
809  * subdomain(i) = triangulation.locally_owned_subdomain();
810  * data_out.add_data_vector(subdomain, "subdomain");
811  *
812  * data_out.build_patches();
813  *
814  * @endcode

816  * The next step is to write this data to disk. We write up to 8 VTU files

818  * generated, which groups the written VTU files.
819  *
820  * @code
821  * data_out.write_vtu_with_pvtu_record(

834  *
835  * The function that controls the overall behavior of the program is again
836  * like the one in @ref step_6 "step-6". The minor difference are the use of

838  * console (see also @ref step_17 "step-17") and that we only generate graphical output if
839  * at most 32 processors are involved. Without this limit, it would be just

907  * <a name="main"></a>
908  * <h4>main()</h4>
909  *
910 
911  *
912  * The final function, <code>main()</code>, again has the same structure as in
913  * all other programs, in particular @ref step_6 "step-6". Like the other programs that use
914  * MPI, we have to initialize and finalize MPI, which is done using the helper
915  * object Utilities::MPI::MPI_InitFinalize. The constructor of that class also
916  * initializes libraries that depend on MPI, such as p4est, PETSc, SLEPc, and
917  * Zoltan (though the last two are not used in this tutorial). The order here
918  * is important: we cannot use any of these libraries until they are
919  * initialized, so it does not make sense to do anything before creating an
920  * instance of Utilities::MPI::MPI_InitFinalize.
921  *
922 
923  *
924  * After the solver finishes, the LaplaceProblem destructor will run followed
925  * by Utilities::MPI::MPI_InitFinalize::~MPI_InitFinalize(). This order is
926  * also important: Utilities::MPI::MPI_InitFinalize::~MPI_InitFinalize() calls
927  * <code>PetscFinalize</code> (and finalization functions for other
928  * libraries), which will delete any in-use PETSc objects. This must be done
929  * after we destruct the Laplace solver to avoid double deletion
930  * errors. Fortunately, due to the order of destructor call rules of C++, we
931  * do not need to worry about any of this: everything happens in the correct
932  * order (i.e., the reverse of the order of construction). The last function
933  * called by Utilities::MPI::MPI_InitFinalize::~MPI_InitFinalize() is
934  * <code>MPI_Finalize</code>: i.e., once this object is destructed the program
935  * should exit since MPI will no longer be available.
936  *
937  * @code

939  * {
940  * try
941  * {

947  * LaplaceProblem<2> laplace_problem_2d;

949  * }
950  * catch (std::exception &exc)

1097 <a name="Possibilitiesforextensions"></a><h3>Possibilities for extensions</h3>
1098 
1099 
1100 从某种意义上说，这个程序是拉普拉斯方程的终极解算器：只要你有足够的处理器，它基本上可以把方程解到你想要的精度。由于拉普拉斯方程本身在这种精度水平上并不十分有趣，因此，更有趣的扩展可能性不在于这个程序，而在于它之后的内容。例如，本教程中的其他几个程序都有相当长的运行时间，特别是在3D中。因此，使用这里解释的技术来扩展其他程序以支持并行的分布式计算将是有趣的。我们在@ref step_32 "step-32"教程程序中对@ref step_31 "step-31"做了这样的处理，但同样的做法也适用于，例如，用于双曲时间相关问题的@ref step_23 "step-23"和@ref step_25 "step-25"，用于气体动力学的@ref step_33 "step-33"，或用于纳维-斯托克斯方程的@ref step_35 "step-35"。
1101 
1102 也许同样有趣的是后处理的问题。如上所述，我们只展示了16个处理器的解决方案和网格的图片，因为4,096个处理器解决10亿个未知数会产生几10G的图形输出。目前，除非在至少几百个处理器上运行，否则没有任何程序能够以任何合理的方式将如此大量的数据可视化。然而，有一些方法，可视化程序直接与每个处理器上的求解器进行通信，每个可视化进程渲染这个处理器上的求解器所计算的场景部分。实现这样的接口将允许快速可视化那些在其他方面不适合用图形显示的东西。
